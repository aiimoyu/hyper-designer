# 可扩展文档格式设计

## 1. 可扩展性需求

### 1.1 业务需求
1. **新增功能**：当项目新增功能时，需要能够方便地在文档中扩展
2. **新增模块**：当项目新增模块时，需要能够方便地在文档中扩展
3. **新增接口**：当项目新增接口时，需要能够方便地在文档中扩展
4. **修改功能**：当项目修改功能时，需要能够方便地在文档中修改

### 1.2 技术需求
1. **模块化**：文档应该模块化，便于独立更新
2. **版本化**：文档应该支持版本管理
3. **格式统一**：文档格式应该统一，便于处理
4. **机器可读**：文档应该同时支持人类可读和机器可读

## 2. 文档结构设计

### 2.1 YAML Front Matter
每个Markdown文档都包含YAML Front Matter，用于存储元数据：
```yaml
---
title: 文档标题
version: 1.0
last_updated: YYYY-MM-DD
type: document_type
sections:
  - section1
  - section2
metadata:
  key: value
---
```

### 2.2 模块化章节
每个文档由多个独立的章节组成，每个章节可以独立更新：
```markdown
# 文档标题

## 章节1
[章节1内容]

## 章节2
[章节2内容]

## 章节3
[章节3内容]
```

### 2.3 引用机制
文档之间通过引用机制关联，便于维护一致性：
```markdown
## 功能说明
详见 [功能树](function-tree.md#功能F001)

## 模块说明
详见 [模块关系](module-relationships.md#模块M001)
```

## 3. 扩展机制

### 3.1 新增功能扩展
当新增功能时，在功能树中添加新的功能节点：
```markdown
## 功能层次结构
```mermaid
graph TD
  root[系统功能] --> f1[功能1]
  root --> f2[功能2]
  root --> f3[新功能]  # 新增功能
```

## 功能说明
| 功能ID | 功能名称 | 描述 | 父功能 | 子功能 |
|--------|----------|------|--------|--------|
| F001 | 功能1 | 描述 | - | - |
| F002 | 功能2 | 描述 | - | - |
| F003 | 新功能 | 描述 | - | - |  # 新增功能
```

### 3.2 新增模块扩展
当新增模块时，在模块关系中添加新的模块节点：
```markdown
## 模块清单
| 模块ID | 模块名称 | 描述 | 路径 | 类型 |
|--------|----------|------|------|------|
| M001 | 模块1 | 描述 | src/module1 | 核心 |
| M002 | 模块2 | 描述 | src/module2 | 辅助 |
| M003 | 新模块 | 描述 | src/new-module | 核心 |  # 新增模块

## 模块依赖关系
```mermaid
graph LR
  m1[模块1] --> m2[模块2]
  m2 --> m3[新模块]  # 新增依赖
```
```

### 3.3 新增接口扩展
当新增接口时，在接口契约中添加新的接口定义：
```markdown
## API 清单
| API ID | API 名称 | 类型 | 模块 | 路径 | 签名 |
|--------|----------|------|------|------|------|
| A001 | API1 | REST | M001 | /api/v1/resource1 | GET /api/v1/resource1 |
| A002 | API2 | REST | M002 | /api/v1/resource2 | POST /api/v1/resource2 |
| A003 | 新API | REST | M003 | /api/v1/resource3 | GET /api/v1/resource3 |  # 新增接口
```

### 3.4 修改功能扩展
当修改功能时，更新功能树中的相应节点：
```markdown
## 功能说明
| 功能ID | 功能名称 | 描述 | 父功能 | 子功能 |
|--------|----------|------|--------|--------|
| F001 | 功能1 | 更新后的描述 | - | F001.1 |  # 修改功能 |
| F002 | 功能2 | 描述 | - | - |
```

## 4. 版本管理

### 4.1 版本号规则
1. **主版本号**：当文档格式发生重大变化时
2. **次版本号**：当新增章节时
3. **修订号**：当修改内容时

### 4.2 版本历史
```yaml
version_history:
  - version: "1.0.0"
    date: "2026-03-22"
    changes: "初始版本"
  - version: "1.0.1"
    date: "2026-03-23"
    changes: "新增功能F003"
  - version: "1.1.0"
    date: "2026-03-24"
    changes: "新增模块M003"
```

### 4.3 兼容性
1. **向后兼容**：新版本应该兼容旧版本
2. **迁移工具**：提供迁移工具，将旧版本迁移到新版本

## 5. 格式规范

### 5.1 Markdown格式规范
1. **标题层级**：使用标准的Markdown标题层级
2. **表格格式**：使用标准的Markdown表格格式
3. **代码块**：使用标准的Markdown代码块格式
4. **引用格式**：使用标准的Markdown引用格式

### 5.2 YAML格式规范
1. **缩进**：使用2个空格缩进
2. **键值对**：使用标准的YAML键值对格式
3. **列表**：使用标准的YAML列表格式
4. **注释**：使用标准的YAML注释格式

### 5.3 Mermaid格式规范
1. **图表类型**：使用标准的Mermaid图表类型
2. **节点命名**：使用中文，简洁明了
3. **边命名**：使用中文，描述关系
4. **子图命名**：使用中文，描述模块或功能组

## 6. 扩展示例

### 6.1 新增功能示例
假设项目新增了"用户权限管理"功能：

**步骤1：更新功能树**
```markdown
## 功能层次结构
```mermaid
graph TD
  root[系统功能] --> core[核心功能]
  root --> aux[辅助功能]
  
  core --> f1[用户管理]
  core --> f2[数据处理]
  core --> f3[权限管理]  # 新增功能
  
  f1 --> f11[用户注册]
  f1 --> f12[用户登录]
  f1 --> f13[用户信息管理]
  
  f3 --> f31[角色管理]  # 新增子功能
  f3 --> f32[权限分配]  # 新增子功能
```

## 功能说明
| 功能ID | 功能名称 | 描述 | 父功能 | 子功能 |
|--------|----------|------|--------|--------|
| F001 | 用户管理 | 管理用户信息 | - | F001.1, F001.2, F001.3 |
| F003 | 权限管理 | 管理用户权限 | - | F003.1, F003.2 |  # 新增功能 |
| F003.1 | 角色管理 | 管理用户角色 | F003 | - |  # 新增子功能 |
| F003.2 | 权限分配 | 分配用户权限 | F003 | - |  # 新增子功能 |
```

**步骤2：更新模块关系**
```markdown
## 模块清单
| 模块ID | 模块名称 | 描述 | 路径 | 类型 |
|--------|----------|------|------|------|
| M001 | 用户管理模块 | 管理用户信息 | src/modules/user | 核心 |
| M003 | 权限管理模块 | 管理用户权限 | src/modules/permission | 核心 |  # 新增模块 |

## 模块依赖关系
```mermaid
graph LR
  um[用户管理模块] --> db[数据库访问模块]
  pm[权限管理模块] --> db  # 新增依赖
  pm --> um  # 新增依赖
```
```

**步骤3：更新接口契约**
```markdown
## API 清单
| API ID | API 名称 | 类型 | 模块 | 路径 | 签名 |
|--------|----------|------|------|------|------|
| A001 | 用户注册 | REST | M001 | /api/v1/users | POST /api/v1/users |
| A004 | 角色管理 | REST | M003 | /api/v1/roles | GET /api/v1/roles |  # 新增接口 |
| A005 | 权限分配 | REST | M003 | /api/v1/permissions | POST /api/v1/permissions |  # 新增接口 |
```

### 6.2 新增模块示例
假设项目新增了"消息队列"模块：

**步骤1：更新模块关系**
```markdown
## 模块清单
| 模块ID | 模块名称 | 描述 | 路径 | 类型 |
|--------|----------|------|------|------|
| M001 | 用户管理模块 | 管理用户信息 | src/modules/user | 核心 |
| M004 | 消息队列模块 | 处理异步消息 | src/modules/mq | 基础 |  # 新增模块 |

## 模块依赖关系
```mermaid
graph LR
  um[用户管理模块] --> mq[消息队列模块]  # 新增依赖
  dp[数据处理模块] --> mq  # 新增依赖
```
```

**步骤2：更新数据流**
```markdown
## 数据流图
```mermaid
graph LR
  user[用户] --> um[用户管理模块]
  um --> mq[消息队列模块]  # 新增数据流
  mq --> dp[数据处理模块]  # 新增数据流
```

## 数据流说明
| 来源 | 目标 | 数据 | 描述 |
|------|------|------|------|
| 用户管理模块 | 消息队列模块 | 用户操作消息 | 用户操作产生的消息 |  # 新增数据流 |
| 消息队列模块 | 数据处理模块 | 处理任务 | 异步处理任务 |  # 新增数据流 |
```

## 7. 维护指南

### 7.1 新增内容
1. 在相应的文档中添加新的内容
2. 更新YAML Front Matter中的版本号和最后更新时间
3. 更新相关的图表
4. 验证文档格式正确

### 7.2 修改内容
1. 在相应的文档中修改内容
2. 更新YAML Front Matter中的版本号和最后更新时间
3. 更新相关的图表
4. 验证文档格式正确

### 7.3 删除内容
1. 在相应的文档中删除内容
2. 更新YAML Front Matter中的版本号和最后更新时间
3. 更新相关的图表
4. 验证文档格式正确
