# Phase 1 — 需求场景分析

## 目标

将原始需求输入（GitHub Issue、口述、功能请求等）转化为结构化的**需求分析说明书**，明确功能边界、优先级、性能规格和验收标准。

---

## 输入处理策略

**无论输入是否充分，都必须与用户交互确认理解后再输出文档。**

### 批量提问 + 场景确认

**一次性提出所有关键问题**（不超过 4 个），等用户回答后再进行场景确认：

```
为了准确分析需求，我有几个关键问题：

1. 这个功能解决了什么具体痛点？现有方式有什么不足？
2. 主要使用者是谁？他们最典型的操作流程是什么？
3. 哪些功能是绝对不能少的（P0），哪些是锦上添花？
4. 有没有明确的性能要求、技术约束、或已知的参考实现？
```

收到回答后，立即进行场景确认：

```
我理解这个需求的核心场景是：

**典型场景**：
[用 2-4 句话描述：谁 / 在什么情况下 / 做了什么 / 系统如何响应 / 最终结果]

这个理解正确吗？有需要补充或调整的场景吗？确认后我将输出正式文档。
```

### §2 项目背景与 5W2H 分析

#### 2.1 项目背景

**需求价值**（1-2 句）：该需求为业务带来的核心价值，说明"为什么值得做"。追溯到业务根源，避免泛泛而谈（❌"提升用户体验"→ ✅"将故障发现时间从 15 分钟缩短至实时"）。

**需求描述**（1-2 句）：需求的本质和范围，说明"要做什么"。

#### 2.2 5W2H 分析

每项限 1-2 句，聚焦核心信息，**避免抽象描述**：

| 维度 | 说明要点 | 常见错误 |
|------|---------|---------|
| **What** | 核心功能内容，具体可验证 | ❌"支持用户交互" → ✅"在故障模式抽取节点增加用户确认/修正入口" |
| **Why** | 追溯到业务根源和痛点 | ❌"提高效率" → ✅"现有自动生成准确率不足，需人工干预提升质量" |
| **Who** | 主要使用者 + 管理者，涵盖所有关键角色 | 不要遗漏管理员角色 |
| **When** | 使用时机和触发条件 | |
| **Where** | 物理环境、技术平台、系统上下文 | |
| **How** | 用户核心操作流程 | |
| **How Much** | 规模预估、开发资源投入（人月） | 需包含人力和时间 |

### §3 业务功能与场景

#### 3.1 功能清单与优先级

| 功能编号 | 功能名称 | 功能描述 | 优先级 |
|----------|----------|----------|--------|
| F-01 | [名称] | [2-3 句描述核心行为和用户价值，说明包含哪些子能力] | **P0 - Must Have** |
| F-02 | [名称] | [描述] | **P1 - Should Have** |
| F-03 | [名称] | [描述] | **P2 - Nice to Have** |

**优先级定义**：
- **P0 Must Have**：缺少则需求无法完成，必须实现
- **P1 Should Have**：核心增值功能，计划实现
- **P2 Nice to Have**：锦上添花，有资源再做

#### 3.2 主成功场景

用具名角色（如"运维人员张三"）描述最典型的端到端操作流：

```
[角色] 遇到 [触发事件]
→ [角色] 执行 [操作A]
→ 系统 [响应B]
→ [角色] 查看 / 确认 / 修正
→ 系统 [根据输入生成结果C]
→ [角色] 完成目标
→ 系统记录 [后续影响/知识沉淀]
```

#### 3.3 业务场景分类

| 类型 | 定义 | 本需求中的例子 |
|------|------|--------------|
| **业务场景** | 直接产生业务价值的核心操作 | [举例] |
| **操作场景** | 日常使用的辅助操作（搜索、筛选、导出） | [举例] |
| **维护场景** | 管理员配置和运维操作 | [举例] |
| **制造场景** | 系统批量生成或处理内容 | [举例] |

### §4 性能规格（重要）

**性能规格是需求分析的核心内容之一**，必须在需求阶段明确量化指标，为后续设计和验收提供依据。

#### 4.1 响应时间要求

| 操作类型 | 响应时间目标 | 测量条件 | 说明 |
|----------|-------------|----------|------|
| [关键交互操作] | P95 < [N]ms, P99 < [M]ms | [并发用户数、数据量] | [如：用户点击查询后到结果展示] |
| [后台任务] | < [N]s | [数据规模] | [如：批量导入 1000 条记录] |
| [页面加载] | 首屏 < [N]s | [网络条件] | [如：首次内容渲染时间] |

**响应时间分解示例**：

```
端到端响应时间目标：P95 < 500ms

分解：
┌─────────────────┬──────────┬────────────────────────────┐
│ 处理环节         │ 时间分配  │ 说明                        │
├─────────────────┼──────────┼────────────────────────────┤
│ 网络传输         │ 50ms     │ 同机房内网延迟               │
│ API 网关         │ 30ms     │ 认证 + 路由                  │
│ 业务处理         │ 200ms    │ 核心业务逻辑                 │
│ 数据库查询       │ 100ms    │ 含索引优化                   │
│ 缓存操作         │ 20ms     │ Redis 读写                  │
│ 设计余量         │ 100ms    │ 20% 余量应对峰值             │
├─────────────────┼──────────┼────────────────────────────┤
│ 合计             │ 500ms    │ 满足目标                     │
└─────────────────┴──────────┴────────────────────────────┘
```

### §5 验收方法

#### 5.1 验收标准

| 验收项 | 验收标准 | 验收方法 | 优先级 |
|--------|----------|----------|--------|
| [功能验收] | [可量化标准，含基线说明] | [具体测试方法] | P0 |
| [性能验收] | [响应时间/吞吐量指标] | [性能测试工具和方法] | P0 |
| [安全验收] | [安全测试通过条件] | [安全测试方法] | P0 |

**验收标准写作要点**：
- 标准必须**可量化**（有数字、有基线、有测试方法）
- P0 验收项阻断发布，P1/P2 可延期
- 性能验收必须包含 P95/P99 分位值

**验收标签示例**：

```
❌ 错误示例：
- "系统响应要快"
- "用户体验要好"
- "性能满足要求"

✅ 正确示例：
- "查询接口 P95 响应时间 < 500ms，P99 < 1000ms（测试条件：100 并发用户，数据量 10 万条）"
- "批量导入 1000 条记录完成时间 < 30s"
- "经反馈修正后准确率较基线提升 ≥20%；基线=现有无反馈机制的结果"
```

#### 5.2 验收流程

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  单元测试    │ ──→ │  集成测试    │ ──→ │  系统测试    │ ──→ │  验收测试    │
│  (开发者)    │     │  (测试人员)  │     │  (测试人员)  │     │  (用户参与)  │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
      │                   │                   │                   │
      ▼                   ▼                   ▼                   ▼
  代码覆盖率          接口联调验证         端到端场景          真实业务场景
  分支覆盖 > 80%      异常路径覆盖        性能基准测试        用户签字确认
```

**各阶段验收内容**：

| 阶段 | 验收内容 | 通过标准 | 负责人 |
|------|----------|----------|--------|
| 单元测试 | 各模块功能正确性 | 覆盖率 > 80%，全部通过 | 开发者 |
| 集成测试 | 模块间接口正确性 | 接口契约验证通过 | 测试人员 |
| 系统测试 | 端到端场景 + 性能 | 所有 P0 场景通过，性能达标 | 测试人员 |
| 验收测试 | 真实业务场景验证 | 用户确认功能满足需求 | 用户 + 产品 |

#### 5.3 测试场景

**每个 P0 功能必须定义至少 2 个测试场景：正常路径 + 异常路径**

| 场景编号 | 场景名称 | 场景类型 | 前置条件 | 操作步骤 | 预期结果 |
|----------|----------|----------|----------|----------|----------|
| TC-001 | [正常路径场景] | 正常 | [初始状态] | 1. [步骤1]<br>2. [步骤2]<br>3. [步骤3] | [可观测的预期结果] |
| TC-002 | [异常路径场景] | 异常 | [异常初始状态] | 1. [触发异常条件] | [优雅失败 + 错误提示] |
| TC-003 | [边界条件场景] | 边界 | [边界状态] | 1. [边界操作] | [正确处理边界情况] |

**测试场景示例**：

```markdown
| 场景编号 | 场景名称 | 场景类型 | 前置条件 | 操作步骤 | 预期结果 |
|----------|----------|----------|----------|----------|----------|
| TC-001 | 正常登录 | 正常 | 用户已注册，账号正常 | 1. 输入正确用户名密码<br>2. 点击登录 | 登录成功，跳转首页 |
| TC-002 | 密码错误 | 异常 | 用户已注册 | 1. 输入错误密码<br>2. 点击登录 | 提示"密码错误"，剩余尝试次数显示 |
| TC-003 | 连续失败锁定 | 异常 | 用户已连续失败4次 | 1. 再次输入错误密码 | 账号锁定15分钟，发送安全提醒 |
```

## 质量自检清单

### 功能完整性
- [ ] 功能清单中每条都有优先级（P0/P1/P2）
- [ ] 主成功场景是具体操作流，不是抽象描述
- [ ] 业务场景分类完整（业务/操作/维护/制造）

### 性能规格
- [ ] 响应时间有 P95/P99 分位值
- [ ] 吞吐量有具体数值和测量条件
- [ ] 资源约束有具体规格（CPU/内存/存储）

### 验收方法
- [ ] 验收标准有量化指标和测试方法
- [ ] 每个 P0 功能有至少 2 个测试场景
- [ ] 验收流程各阶段责任明确

### 约束与边界
- [ ] 技术约束包含具体的框架版本和接口协议
- [ ] Out of Scope 列出了关键排除项
- [ ] 所有内容基于用户提供的信息，无无据假设
