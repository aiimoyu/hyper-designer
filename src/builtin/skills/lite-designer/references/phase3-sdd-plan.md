# Phase 3 — SDD 开发计划

## 目标

将需求设计说明书中的 AR 分配拆解为**可直接执行的开发任务**，生成结构化的开发计划，使开发 Agent（如 Claude Code）能够无歧义地理解任务内容、执行顺序和验收条件。

---

## 前置条件

- 已完成 Phase 2，需求设计说明书已确认
- 所有 AR 的功能点已明确
- 修改围栏已确认

---

## ⚠️ 核心原则：计划划分与排序

### 原则 1：框架优先，渐进细化

**开发计划必须按照依赖关系和技术栈层次进行排序，而非机械按 AR 划分。**

对于全新项目（greenfield），必须遵循以下顺序：

```
Wave 0（基础设施层）：
├── 项目脚手架初始化
├── 构建工具配置（tsconfig, vite/webpack, eslint 等）
├── 目录结构创建
├── 基础类型定义
└── 核心依赖安装

Wave 1（数据与接口层）：
├── 数据模型/Schema 定义
├── 接口类型声明
├── 基础工具函数
└── 配置文件

Wave 2（核心业务层）：
├── 核心业务逻辑实现
├── 服务层实现
└── 业务规则处理

Wave 3（接口与集成层）：
├── API 端点实现
├── UI 组件实现
└── 模块间集成

Wave FINAL（验收层）：
├── 集成测试
├── 端到端测试
└── 需求覆盖审查
```

**对于增量项目（brownfield）**，Wave 0 可能已存在，但仍需确认基础设施是否需要扩展。

### 原则 2：工作量适中，避免极端

**开发计划数量建议**：

| 项目规模 | 建议计划数量 | 单个计划工作量 |
|----------|--------------|----------------|
| 小型（单模块增强） | 1 个 | 1-3 人日 |
| 中型（多模块） | 2 个 | 2-5 人日 |
| 大型（全新系统） | 3 个 | 3-7 人日 |

**核心约束**：**一个开发计划最多 3 个**，超过 3 个应合并或重新评估范围。

**避免极端**：
- ❌ 过少：一个计划包含 10+ 个 AR，工作量超过 2 周
- ❌ 过多：每个 AR 单独一个计划，导致计划碎片化
- ✅ 适中：每个计划包含 2-4 个相关 AR，工作量 2-5 人日

### 原则 3：耦合度优先于 AR 边界

**不要机械按 AR 划分计划，要考虑功能耦合度。**

**合并规则**（应该合并为一个计划）：
- 多个 AR 属于同一模块且共享数据模型
- 多个 AR 有强依赖关系，分开实现会导致反复修改
- 多个 AR 的实现涉及同一组文件
- 多个 AR 共同完成一个完整的用户场景

**拆分规则**（应该拆分为多个计划）：
- AR 涉及完全独立的模块
- AR 可以独立测试和验收
- AR 的实现不需要等待其他 AR 完成
- 单个计划工作量超过 5 人日

**示例**：

```
❌ 错误划分（机械按 AR）：
- 计划 1：AR-001 用户注册
- 计划 2：AR-002 用户登录
- 计划 3：AR-003 密码重置
（问题：三个 AR 共享用户模块，分开实现会导致重复工作）

✅ 正确划分（按耦合度）：
- 计划 1：用户认证模块（包含 AR-001, AR-002, AR-003）
  理由：共享用户数据模型、认证服务、安全策略
```

### 原则 4：全新项目的框架构建计划

**对于全新项目，第一个计划必须是框架构建计划。**

框架构建计划必须包含：

```
## 框架构建计划（Wave 0）

**目标**：搭建项目骨架，为后续开发提供基础支撑。

**必须包含**：
1. 项目初始化
   - 包管理器配置（package.json / pnpm-workspace.yaml）
   - TypeScript 配置（tsconfig.json, tsconfig.build.json）
   - 构建工具配置（vite.config.ts / webpack.config.ts）
   - 代码规范配置（eslint, prettier）

2. 目录结构
   - src/
   - tests/
   - docs/
   - 配置文件目录

3. 基础依赖
   - 运行时依赖
   - 开发依赖
   - 类型定义

4. 开发脚本
   - npm run dev
   - npm run build
   - npm run test
   - npm run lint

**验收标准**：
- [ ] 项目可成功编译（npm run build）
- [ ] 测试框架可运行（npm run test）
- [ ] 代码规范检查通过（npm run lint）
- [ ] 开发服务器可启动（npm run dev）
```

**框架构建计划完成后，才能开始业务功能计划。**

### §5 测试计划（TDD 驱动）

> **核心原则**：测试先行，每个任务必须先定义测试场景，再实现功能。验收通过 = 所有测试通过。

#### 5.1 单元测试规范

**每个开发任务必须包含对应的单元测试文件**，测试文件与源文件遵循以下规范：

| 规范项 | 要求 |
|--------|------|
| **测试文件命名** | `{源文件名}.test.ts` 或 `{源文件名}.spec.ts` |
| **测试文件位置** | 与源文件同目录，或 `__tests__/` 目录镜像结构 |
| **测试框架** | 项目已配置的测试框架（Vitest / Jest / Mocha） |
| **覆盖率要求** | 行覆盖率 > 80%，分支覆盖率 > 70% |

**单元测试结构模板**：

```typescript
describe('{模块名}/{函数名}', () => {
  // ========== 正常路径 ==========
  describe('正常路径', () => {
    it('should [期望行为] when [输入条件]', () => {
      // Arrange: 准备测试数据
      const input = { /* 测试输入 */ };
      const expected = { /* 期望输出 */ };
      
      // Act: 执行被测函数
      const actual = functionUnderTest(input);
      
      // Assert: 验证结果
      expect(actual).toEqual(expected);
    });
  });

  // ========== 异常路径 ==========
  describe('异常路径', () => {
    it('should throw error when [异常条件]', () => {
      const invalidInput = { /* 无效输入 */ };
      
      expect(() => functionUnderTest(invalidInput))
        .toThrow(ExpectedErrorType);
    });
  });

  // ========== 边界条件 ==========
  describe('边界条件', () => {
    it('should handle [边界情况: 空值/最大值/最小值]', () => {
      // 测试边界值
    });
  });
});
```

**单元测试必须覆盖的场景类型**：

| 场景类型 | 覆盖要求 | 示例 |
|----------|----------|------|
| 正常路径 | 至少 1 个 | 有效输入返回正确输出 |
| 异常路径 | 至少 1 个 | 无效输入抛出正确错误 |
| 边界条件 | 按需覆盖 | 空数组、最大值、null/undefined |
| Mock 外部依赖 | 必须隔离 | 数据库、API、文件系统 |

**测试任务清单格式**：

每个任务在"验收条件"中必须包含测试清单：

```markdown
**测试清单**：
- [ ] 测试文件已创建：`src/services/feedback/__tests__/submitFeedback.test.ts`
- [ ] 正常路径：有效反馈提交成功
- [ ] 异常路径：空内容提交失败
- [ ] 异常路径：超长内容截断处理
- [ ] Mock：数据库写入已 mock
- [ ] 运行 `npm test -- src/services/feedback` → PASS
```

### §6 验收标准（QA 场景）

> **验收原则**：每个任务必须定义可自动执行的 QA 场景。验收通过 = 所有场景断言成功。

#### 6.1 QA 场景定义规范

**每个任务必须包含至少 2 个 QA 场景**（正常路径 + 异常路径），格式如下：

```
场景：{场景编号} — {场景名称}
├── 类型：[正常路径 / 异常路径 / 边界条件]
├── 工具：[单元测试 / curl / API 测试 / E2E]
├── 前置条件：[具体初始状态，可执行命令]
├── 步骤：
│   1. [具体操作，含确切命令/函数调用]
│   2. [中间状态验证]
│   3. [断言 — 具体期望值]
├── 断言：
│   - expect(actual).toEqual(expected)
│   - expect(status).toBe(200)
├── 验收命令：[可执行的测试命令]
└── 证据：evidence/{任务编号}-{场景编号}.{ext}
```

#### 6.4 验收失败处理

**当验收失败时，Agent 必须执行以下流程**：

```
验收失败处理流程：
1. 记录失败信息
   - 失败的测试用例名称
   - 失败的断言详情
   - 错误堆栈信息

2. 分析失败原因
   - 实现代码缺陷 → 修复代码
   - 测试用例错误 → 修正测试
   - 设计理解偏差 → 回到设计文档确认

3. 修复后重新验收
   - 重新执行失败的验收命令
   - 确认所有断言通过

4. 更新证据文件
   - 保存新的测试输出到 evidence/
```
