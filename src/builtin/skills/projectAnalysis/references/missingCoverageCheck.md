# missingCoverageCheck Reference

## Current Phase: Missing Coverage Check

### 1. 执行标准

**核心方法论**：

本阶段是严格的诊断检查阶段，用于验证分析覆盖率并输出结构化报告。

**关键说明**：这不是工作流门控检查。

本阶段的输出用于：
- 覆盖率报告和文档
- 测试断言和验证
- 修复建议

工作流不将这些检查作为门控强制执行。这是一个诊断工具，而不是阻塞性验证。

---

### 2. 严格的覆盖率类别

对以下 7 个严格的覆盖率类别执行验证：

#### 1. 缺失组件（Missing Components）

验证组件清单中的所有组件都有对应的分析输出。

**检查方法：**
- 遍历组件清单中的每个组件
- 检查是否存在对应的 Markdown 分析文件
- 标记清单中有但分析输出缺失的组件

**检查项：**
- [ ] 组件清单中的每个组件都有 `{componentSlug}.md` 分析文件
- [ ] 组件名称在清单和分析文件之间匹配

**严重性**：HIGH

#### 2. 缺失文件（Missing Files）

验证组件分析中引用的所有源文件在代码库中存在。

**检查方法：**
- 遍历组件分析中的每个文件引用
- 验证文件在目标项目根目录下存在
- 尊重排除模式并标记不存在引用

**检查项：**
- [ ] 组件分析中引用的源文件存在
- [ ] 文件路径相对于目标项目根目录

**严重性**：MEDIUM

#### 3. 缺失文件夹（Missing Folders）

验证目标项目中是否存在预期的目录结构。

**检查方法：**
- 检查标准项目目录（如 src/、tests/、docs/）
- 验证分析中引用的模块目录和产物目录存在

**检查项：**
- [ ] 标准项目目录存在（src/、tests/、docs/）
- [ ] 分析中引用的模块特定目录存在

**严重性**：LOW

#### 4. API 识别遗漏（Missed API Identification）

识别代码中存在但未在 API 清单中记录的 API。

**检查方法：**
- 从代码中提取所有公开 API
- 与 API 清单交叉比对并检查签名一致性

**检查项：**
- [ ] 代码中的所有公开 API 都已在 API 清单中记录
- [ ] API 签名在代码和清单之间匹配

**严重性**：HIGH

#### 5. Mermaid 覆盖不足（Insufficient Mermaid Coverage）

验证关键图表在架构文档和组件分析中存在且完整。

**检查方法：**
- 检查系统架构图、关键交互图、数据流图是否覆盖关键路径
- 验证 Mermaid 语法有效

**检查项：**
- [ ] 架构文档中存在系统架构图
- [ ] 关键流程存在组件交互图
- [ ] 组件分析中的序列图覆盖关键用例
- [ ] Mermaid 语法有效

**严重性**：MEDIUM

#### 6. 交叉引用损坏（Broken Cross References）

验证项目分析文档之间的所有内部引用有效。

**检查方法：**
- 遍历所有文档中的内部链接与对象引用
- 验证组件 slug / API slug / 文件路径可解析

**检查项：**
- [ ] 架构文档与组件分析之间的内部链接可解析
- [ ] 组件引用指向有效的 componentSlug
- [ ] API 引用指向 API 清单中的有效 slug

**严重性**：MEDIUM

#### 7. 系统/组件不一致（System/Component Inconsistency）

验证架构文档与组件分析之间的对齐。

**检查方法：**
- 遍历组件清单并加载对应组件分析
- 对比名称、依赖、API映射与维度结论的一致性

**检查项：**
- [ ] 组件名称在清单和分析之间匹配
- [ ] 清单中的组件依赖关系反映在分析中
- [ ] API 清单引用与组件分析一致

**严重性**：HIGH

### 3. 输出

**覆盖率报告**：`./.hyper-designer/projectAnalysis/coverage-report.md`

纯 Markdown 格式的覆盖率报告，包含：
- 摘要部分（总检查数、通过数、失败数、警告数）
- 7 个覆盖率类别的各自部分
- 每个问题标记严重性级别：[HIGH]、[MEDIUM] 或 [LOW]
- 缺失或损坏内容的清晰描述
- 受影响工件的引用（文件、组件、图表）

### 5. 判定逻辑

根据检查结果生成总体判定：

1. **PASSED**：所有 7 个类别的检查均无 HIGH 严重性问题
2. **FAILED**：任何类别存在 HIGH 严重性问题
3. **WARNING**：存在 MEDIUM 或 LOW 严重性问题，但无 HIGH 严重性问题

---

### 6. 严重性级别定义

- **HIGH（高）**：阻止分析完整性或导致关键数据不一致
- **MEDIUM（中）**：损害分析质量或会引入明显技术债
- **LOW（低）**：建议改进项，不影响当前分析闭环

### 7. 修复建议模式

- **立即修复（Immediate）**：必须马上修复的问题，通常对应 HIGH
- **短期改进（Short-term）**：下次迭代应修复，通常对应 MEDIUM
- **长期方向（Long-term）**：持续优化项，通常对应 LOW
