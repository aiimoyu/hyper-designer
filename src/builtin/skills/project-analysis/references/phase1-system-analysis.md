# phase 1: 项目概览分析

## [S1] 初始化 GitNexus

在任何探索或提问之前执行。Load 'references/gitnexus-commands.md' for 命令语法。

```bash
cd <project-path>
npx -y gitnexus@latest analyze --skip-agents-md --skip-git .
```

- 成功：记录节点数、边数、集群数，后续分析可使用 GitNexus 辅助导航
- 失败：记录失败原因，后续使用文件系统工具（find/grep）替代，并在文档中注明

---

## [S2] 探索项目结构

目标：建立项目物理结构认知。此步骤不读取 README/docs 等文档文件——它们可能过时，配置文件和代码结构才是事实。

### [A2.1] 物理结构探索

示例命令：

```bash
# 目录结构（3 层深度）
find <project-path> -maxdepth 3 -type d
# 技术栈配置文件
cat package.json / go.mod / requirements.txt / Cargo.toml / pom.xml
# 配置与环境文件
ls -la <project-path>/ | grep -E '\.(json|yaml|yml|toml|ini|env)'
```

### [A2.2] GitNexus 导航分析

Load 'references/gitnexus-commands.md' for 完整命令语法。执行以下查询定位关键代码区域：

- 项目节点总数（规模感知）
- 最复杂的执行流程（stepCount 排序）
- 被最多文件依赖的核心文件
- 自然模块社区

示例命令：

```bash
# 项目规模
npx gitnexus cypher "MATCH (n) RETURN count(n) AS total" --repo <repo>
# 最核心的执行流程（步骤最多 = 最复杂的路径）
npx gitnexus cypher "MATCH (p:Process) RETURN p.heuristicLabel, p.stepCount ORDER BY p.stepCount DESC LIMIT 8" --repo <repo>
# 最核心的文件（被最多文件依赖）
npx gitnexus cypher "MATCH (f:File)<-[:CodeRelation {type: 'IMPORTS'}]-(g:File) RETURN f.name, count(g) AS deps ORDER BY deps DESC LIMIT 10" --repo <repo>
# 自然模块社区
npx gitnexus cypher "MATCH (c:Community) RETURN c.heuristicLabel, c.symbolCount ORDER BY c.symbolCount DESC LIMIT 12" --repo <repo>
```

此时你拥有的是"地图"，还没有"实地考察"。继续下一步。

## [S3] 深度阅读关键代码

这是本阶段最重要的步骤。GitNexus 告诉你"去哪里看"，这一步你要"实际去看"。

- **多Subagent并行**：同时委派多个 Subagent 执行以下任务。
- **传入gitnexus使用方法**：Subagent 不具备使用 GitNexus 的能力，因此你需要在提示词中明确 Gitnexus 的使用方式或示例。

| Subagent | 任务名称 |
|----------|----------|
| A3.1 | 入口与配置分析 |
| A3.2 | 核心业务流程追踪 |
| A3.3 | 系统边界与横切关注点 |
| A3.4 | 模块结构与依赖分析 |


---

### [A3.1] 入口与配置分析

**读取目标**：入口文件、包管理配置、环境变量文件、构建配置、CI/CD 配置、Dockerfile。

**理解要点**：

- 项目如何初始化、启动命令是什么
- 依赖注入 / 模块加载顺序
- 配置加载方式（环境变量、配置文件、命令行参数）
- 技术栈各组件的确切版本
- 环境变量清单及其来源文件
- 部署方式与流水线结构

---

### [A3.2] 核心业务流程追踪

**读取目标**：从 S2 识别的入口点出发，沿调用链追踪 3–5 条最重要的业务流程。

**理解要点**：

- 每条流程的触发条件、参与模块、关键决策点
- 流程中的状态流转（有无显式状态机或隐式状态变化）
- 数据在各模块间的变换过程（输入形态 → 中间形态 → 持久化形态 → 输出形态）
- 全局状态的管理策略（内存 / Store / 数据库 / 缓存）

---

### [A3.3] 系统边界与横切关注点

**读取目标**：外部服务集成代码、中间件 / 拦截器、错误处理、日志、认证鉴权、插件机制相关文件。

**理解要点**：

- 系统与哪些外部用户 / 系统 / 服务交互，集成方式是什么
- 代码实际分成几层，每层的职责边界和违规现象
- 错误处理、日志、认证鉴权、数据校验的统一方案
- 是否存在插件 / Hook / 中间件管道等扩展机制
- 对外暴露的 API 协议及文档化程度
- 可观察到的架构风险和技术债务

---

### [A3.4] 模块结构与依赖分析

**读取目标**：各模块的入口文件 / index 文件、模块间 import 关系、共享类型定义。结合 GitNexus 的分析和代码文件分析结果。

**理解要点**：

- 模块职责理解（为什么存在，GitNexus 不会告诉你）
- 项目按何种依据划分模块（业务域 / 技术层 / 功能职责）
- 每个模块的职责边界、对应路径、所属架构层
- 模块间的依赖方向和调用关系
- 模块间的通信模式（直接调用 / 事件 / 消息队列 / HTTP/RPC / 共享状态）
- 哪些模块被依赖次数最多（耦合热点）
- 每个模块依赖的外部包 / 服务
- 隐藏的设计决策（为什么这样设计、权衡是什么）

**模块粒度确认**：

模块边界是主观决策，代码无法给出正确答案。生成至少两种粒度方案，询问用户：

- "你更倾向于使用哪种粒度？我推荐___。因为___。"

## [S5] 生成 Architecture.md

此文档不能只反映代码表面结构。每个架构判断都须引用具体代码文件和行号。

- **模板载入**：Load 'assets/architecture-template.md' for 输出结构。
- **缺口识别**：识别 [S3] 的缺失项，对缺失项进行补充阅读
- **实事求是**：DO NOT 进入生成，除非确认全部缺失项已探索，不能凭推测。
- **最终生成**：根据模板生成 Architecture.md 文件

### Architecture自检

- [ ] 系统边界图包含所有已识别的外部参与者和外部系统
- [ ] 分层反映代码的实际组织方式，而非理想架构
- [ ] 跨切面关注点每行都标注了关键文件
- [ ] 入口点表格覆盖了所有入口类型
- [ ] 核心流程 ≥ 3 条，每条包含触发条件、状态流转图、跨模块序列图，且引用了具体文件:行号
- [ ] 状态管理策略基于实际代码实现，非猜测
- [ ] 端到端数据流标注了经过的模块和变换说明
- [ ] 扩展机制和对外 API 边界基于代码证据
- [ ] 架构风险和技术债务来自代码观察，标注了概率 / 严重度 / 优先级

## [S6] 生成 Modules.md

- **模板载入**：Load 'assets/modules-template.md' for 输出结构。
- **缺口识别**：识别 [S3] 的缺失项，对缺失项进行补充阅读
- **慎用子模块**：仅项目较大时拆分子模块，拆分前请拷问自己"父模块是否足够臃肿需要拆分？"
- **用户反馈**：至少提供两种模块粒度方案，询问用户偏好 “你更倾向于使用哪种粒度？我推荐___。因为___。”
- **实事求是**：DO NOT 进入生成，除非确认全部缺失项已探索，不能凭推测。
- **最终生成**：根据模板生成 Modules.md 文件

### Modules自检

- [ ] 模块数量是否合理，是否过度拆分子模块
- [ ] 是否向用户确认模块粒度方案
- [ ] 模块划分说明了依据，并与 Architecture.md 的分层对应
- [ ] 模块层次树和模块清单的 ID 编号一致，ID 在三份文档中全局唯一
- [ ] 模块分层视图与 Architecture.md §2 的分层一致
- [ ] 依赖关系图和依赖矩阵基于实际 import / 调用关系，二者数据一致
- [ ] 外部依赖映射覆盖了所有模块的第三方依赖，标注了版本和风险
- [ ] 耦合热点标注了被依赖次数，并给出了风险等级
- [ ] 通信模式表格覆盖了项目中实际存在的所有通信方式，标注了关键文件

对文档 `Overview.md`、`Architecture.md`、`Modules.md` 进行验证。验证 Subagent 使用独立上下文，不继承生成阶段记忆——这确保验证的客观性，避免生成时的盲点被继承。

- **询问用户**：MUST 询问用户是否需要跳过验证，因为该阶段较长，若用户说可以跳过则不执行
