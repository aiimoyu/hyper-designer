# phase 1: 模块详细分析

**输入依赖**： Overview.md、Architecture.md、Modules.md

**质量标准**： 每个模块详情文件达到"新开发者读完后能独立修改该模块"的水平。

**基于先前分析**：NEVER 在无 Modules.md 情况下推进分析，如无法载入该文件，必须向用户提示并停止。


---

## [S1] 评估分析深度与范围确认

不是所有模块都需要同等深度的分析。根据上一步输出件 `Modules.md` 中的依赖关系和模块职责进行分级。

**模块扁平分析**：如果 Modules.md 中存在子模块，你需要扁平化的对所有模块进行分析，而不是嵌套子模块。

### 优先级分类标准

| 优先级 | 标准 | 分析深度 |
|--------|------|----------|
| 高 | 被最多模块依赖 / 含核心业务逻辑 / 修改频率高 | 完整分析：所有函数签名、完整流程追踪、设计模式识别 |
| 中 | 适度被依赖 / 专门领域逻辑 | 标准分析：公共接口完整、关键流程、省略次要内部函数 |
| 低 | 工具类 / 配置 / 很少修改 | 简化分析：接口契约与依赖关系为主，省略内部实现细节 |

### 执行要求

1. 从 Modules.md 提取模块清单、路径、依赖关系
2. 按上述标准为每个模块分配优先级
3. 若模块数量 > 8，向用户展示优先级分类结果，确认是否全部分析还是先处理高优先级模块
4. **在分析范围确认前不启动任何 Subagent**

## [S2] 并行生成模块文档

同时委派多个 Subagent，对确认范围内的每个模块单独的进行深入分析。所有模块同时启动，不串行等待。

### Subagent 任务模板

对每个模块，使用以下模板生成 Subagent 任务描述：

```
任务：深入分析模块 {ModuleID}-{ModuleName} 并生成模块详情文件

上下文：
  项目路径：{project-path}
  模块路径：{module_path}
  优先级：{高/中/低}
  所属层：{layer}
  依赖于：{dependencies}（来自 Modules.md）
  被依赖于：{dependents}（来自 Modules.md）

输出文件：[输出目录]/modules/{ModuleID}-{ModuleName}.md

执行步骤：

1. 代码深度阅读（必须）
   阅读模块路径下的所有文件，建立完整认知。
   辅助以 gitnexus 分析，可参考 `project-analysis` SKILL 中的 `references/gitnexus-commands.md`

   a. 文件结构梳理
      - 列出所有文件，理解文件间的 import 关系
      - 识别入口文件（index / main / 被外部引用最多的文件）

   b. 公开接口分析
      - 识别所有 export（函数、类、类型、常量）
      - 记录完整签名（参数类型、返回类型、泛型约束）
      - 标注每个 export 的文件路径和行号

   c. 内部实现分析
      - 追踪核心调用链（入口 → 处理 → 输出）
      - 识别设计模式并定位代码证据
      - 分析数据流：入站形态 → 中间变换 → 出站形态
      - 记录关键算法和业务策略

   d. 依赖关系验证
      - 检查所有 import / require 语句
      - 区分内部依赖（项目模块）和外部依赖（第三方包）
      - 验证与 Modules.md 中声明的依赖一致

   e. 质量与风险评估
      - 识别代码坏味道（过大类、过长函数、硬编码等）
      - 评估测试覆盖情况
      - 标注潜在风险（未处理异常、并发安全、内存泄漏）

2. 按模板生成文件
   MUST 基于 `project-analysis` SKILL 中 assets中的模板进行生成
   Load 'assets/module-detail-template.md'，逐节填写。确保：
   - 概述准确反映模块在系统中的角色
   - 所有公开接口已列出，签名与源码一致
   - 代码引用包含 文件:行号 格式
   - 设计模式附有代码证据，非推测
   - 关键流程的调用链和时序图引用具体文件:行号
   - 所有路径使用相对路径，无绝对路径

优先级调整：
  - 高优先级：完成模板所有章节
  - 中优先级：可省略"关键算法/策略"和"开发指南"中的扩展指南
  - 低优先级：可省略"内部实现"全节、"关键流程"中的时序图、"代码质量与风险"
```

### 质量要求

| 要求 | 验证方式 |
|------|----------|
| 所有 export 已列出 | `grep -r "export" {module_path}` 结果与文档公共接口章节对比 |
| 代码引用有行号 | 文档中所有代码引用包含 `:{line}` 或 `L{line}` 格式 |
| 设计模式有代码证据 | 每个模式行包含 `path:line` 而非仅文字描述 |
| 关键流程有具体文件链 | 调用链格式为 `file.ts:45 → file.ts:23`，非 "A 调用 B" |
| 依赖与 Modules.md 一致 | 文档中的依赖列表与 Modules.md 声明的依赖匹配 |
| 无绝对路径 | `grep -E "^/|[A-Z]:\\\\" {output_file}` 无结果 |

## [S4] 拆分 principles 文件

目标：遍历 `modules/` 下所有模块文件，读取每个文件最后一个章节 `## 开发指南`，结合 `Architecture.md`、`Modules.md` ，提炼出项目级黄金准则。

- **载入模板**：Load 'assets/principles-template.md' for 输出结构。
- **生成文件**：将所有原则拆分到 `principles/` 文件夹下，并按主题分类成独立 Markdown 文件。

### 文件命名

- 使用 `kebab-case.md`
- 文件名应直接表达主题，例如：
  - `naming-style.md`
  - `error-handling-principle.md`
  - `skill-writing-principle.md`
  - `dependency-principle.md`

### 分类规则

按主题聚类，不按模块机械拆分，如：

- **工程类**：错误处理、日志、测试、配置
- **架构类**：模块边界、依赖方向、扩展点
- **写作类**：文档组织、SKILL 写法、模板使用
- **治理类**：检查清单、变更流程、风险控制

### 归纳规则

1. **代码优先**：原则必须有代码、架构或模块文档证据。
2. **优先合并共性**：多个模块重复出现的规则，优先合并为项目级原则。
3. **区分作用范围**：
   - 只适用于单一模块 → 模块原则
   - 适用于一类模块 → 分类原则
   - 贯穿整个项目 → 项目原则
4. **避免过度泛化**：不要把“某个模块的习惯”直接升级成全局规范。
