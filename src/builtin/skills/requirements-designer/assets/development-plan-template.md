## §5 测试计划

### 5.1 单元测试规范

| 规范项 | 要求 |
|--------|------|
| **测试文件命名** | `{源文件名}.test.ts` 或 `{源文件名}.spec.ts` |
| **测试文件位置** | 与源文件同目录，或 `__tests__/` 目录镜像结构 |
| **测试框架** | 项目已配置的测试框架（Vitest / Jest / Mocha） |
| **覆盖率要求** | 行覆盖率 > 80%，分支覆盖率 > 70% |

**单元测试结构模板**：

```typescript
describe('{模块名}/{函数名}', () => {
  // ========== 正常路径 ==========
  describe('正常路径', () => {
    it('should [期望行为] when [输入条件]', () => {
      // Arrange
      const input = { /* 测试输入 */ };
      const expected = { /* 期望输出 */ };
      // Act
      const actual = functionUnderTest(input);
      // Assert
      expect(actual).toEqual(expected);
    });
  });

  // ========== 异常路径 ==========
  describe('异常路径', () => {
    it('should throw error when [异常条件]', () => {
      const invalidInput = { /* 无效输入 */ };
      expect(() => functionUnderTest(invalidInput))
        .toThrow(ExpectedErrorType);
    });
  });

  // ========== 边界条件 ==========
  describe('边界条件', () => {
    it('should handle [边界情况]', () => {
      // 测试边界值
    });
  });
});
```

## §6 验收标准

### 6.1 验收执行流程

```
┌──────────────────────────────────────────────────────────────────┐
│                        验收执行流程                               │
├──────────────────────────────────────────────────────────────────┤
│  1. 单元测试 → npm test                                         │
│     预期：所有测试通过，覆盖率 > 80%                              │
│                                                                  │
│  2. 集成测试 → npm run test:integration                          │
│     预期：所有集成测试通过                                        │
│                                                                  │
│  3. QA 场景验证 → 逐个执行各任务的验收命令                        │
│     预期：所有断言通过                                            │
│                                                                  │
│  4. 证据收集 → 将测试输出保存到 evidence/ 目录                    │
│     预期：每个场景有对应证据文件                                   │
│                                                                  │
│  5. 覆盖率报告 → npm run test:coverage                           │
│     预期：满足阈值要求                                            │
└──────────────────────────────────────────────────────────────────┘
```

### 6.2 修改围栏检测

> 本节确保实际代码修改严格遵守需求设计说明书中定义的修改围栏，防止越界修改。

**检测方法**：

```bash
# 1. 获取本次修改的文件列表（相对于开发分支创建时）
git diff --name-only main...HEAD

# 2. 对照修改围栏逐项检查
```

**围栏检测清单**：

| 检测项 | 检测方法 | 预期结果 | 实际结果 |
|--------|----------|----------|----------|
| 冻结模块未被修改 | 检查冻结路径是否在修改列表中 | ❌ 不应出现 | ✅ 通过 / ❌ 越界 |
| 新增模块在指定位置 | 检查新增文件路径是否符合设计 | ✅ 路径正确 | ✅ 通过 / ❌ 偏离 |
| 修改模块未超范围 | 检查修改内容是否超出设计范围 | ✅ 范围内 | ✅ 通过 / ❌ 超范围 |
| 条件修改已获审批 | 检查条件修改是否有审批记录 | ✅ 已审批 | ✅ 通过 / ❌ 未审批 |


### 6.3 验收通过标准

```bash
# 0. 修改围栏检测通过（必须最先执行）
# 对照需求设计说明书 §3.2 模块变更明细表逐项检查
# 预期：无越界修改

# 1. 所有单元测试通过
npm test
# 预期：Tests: X passed, 0 failed

# 2. 覆盖率达标
npm run test:coverage
# 预期：All files | X% | Y% | > 80%

# 3. 所有 QA 场景通过
npm test -- [测试文件路径]
# 预期：每个场景断言全部通过

# 4. 集成测试通过
npm run test:integration
# 预期：Tests: X passed, 0 failed
```

### 6.3 验收失败处理

```
验收失败处理流程：
1. 记录失败信息
   - 失败的测试用例名称
   - 失败的断言详情
   - 错误堆栈信息

2. 分析失败原因
   - 实现代码缺陷 → 修复代码
   - 测试用例错误 → 修正测试
   - 设计理解偏差 → 回到设计文档确认

3. 修复后重新验收
   - 重新执行失败的验收命令
   - 确认所有断言通过

4. 更新证据文件
   - 保存新的测试输出到 evidence/
```

## §8 给开发 Agent 的执行说明

> 以下说明直接面向 Claude Code 等开发 Agent：

**开始前必读**：
1. 读取本文档 §2.3 修改围栏提醒，确认可操作的代码范围
2. 按 §2.2 代码阅读清单阅读现有代码，理解当前实现方式
3. 确认 §2.1 环境与依赖确认中的所有前置条件已满足

**TDD 执行规范**：
- **测试先行**：每个任务必须先创建测试文件，定义 QA 场景，再实现功能
- **红-绿-重构**：先写失败的测试 → 写最小实现使测试通过 → 重构优化
- 按任务依赖顺序执行，不要跳过有前置依赖的任务
- 每完成一个任务，运行对应单元测试，全部通过后再继续下一个
- **遇到与设计不符的情况，先暂停并报告，不要自行修改设计方案**
- 新增代码的命名、注释风格遵循现有代码约定

**验收执行规范**：
- **修改围栏检测先行**：所有验收工作开始前，必须先执行 §6.2 修改围栏检测
- 每个任务完成后，执行该任务的验收命令
- 确认所有 QA 场景断言通过
- 保存测试输出到 `evidence/` 目录
- 若验收失败，按 §6.4 验收失败处理流程处理

**完成标志**：
- [ ] 修改围栏检测通过，无越界修改
- [ ] 所有任务的功能点清单全部勾选完成
- [ ] 所有任务的测试清单全部勾选完成
- [ ] 所有 QA 场景断言通过
- [ ] 单元测试覆盖率 > 80%
- [ ] 集成测试用例全部通过
- [ ] 回归测试无新增失败用例
- [ ] 所有证据文件已保存到 `evidence/` 目录
