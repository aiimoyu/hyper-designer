# Reviewer 方法指南

本文档定义文档评审的执行方法和标准流程。评审是确保文档质量的关键环节，采用 **Reviewer模式** 对输出文档进行系统性评估。

## 评审原则

1. **客观公正**: 基于检查表进行评审，避免主观偏见
2. **全面覆盖**: 确保检查表的每一项都已评估
3. **建设性反馈**: 不仅指出问题，还要提供改进建议
4. **严重程度分级**: 区分必须修复的问题和可改进的建议

## 评审流程

### [1] 确定评审目标

明确要评审的文档和对应的检查表：

```
评审文档: {document-path}
对应检查表: references/{stage}-checklist.md
```

| 阶段 | 文档 | 检查表 |
|------|------|--------|
| Stage 1 |需求分析说明书 | references/requirements-analysis-review-checklist.md |
| Stage 2 | 需求设计说明书 | references/requirements-design-review-checklist.md |
| Stage 3 | 开发计划说明书 | references/development-plan-review-checklist.md |

根据阶段，载入对应的检查表（必须）。

### [2] 执行评审

逐项检查文档内容，对照检查表的每一条规则。

**评审维度**:

- 完整性：内容是否完整、不遗漏重要信息
- 准确性：信息是否准确、符合要求
- 一致性：不同部分之间是否一致、符合逻辑
- 可行性：方案是否可行、是否符合资源限制
- 可追溯性：需求是否可追溯、是否有相关文档支持
- 可读性：文档是否清晰、是否符合规范

## 严重程度分类

### ERROR (错误) - 必须修复

这些是严重问题，不修复将导致：

- 文档无法满足其目的
- 后续工作无法进行
- 重大遗漏或错误

**示例**:

- 核心功能需求缺失
- 数据模型主键未定义
- 关键接口未定义
- 里程碑时间缺失

### WARNING (警告) - 建议修复

这些问题会影响文档质量，但不阻止后续工作：

- 信息不够完整
- 描述不够清晰
- 格式不够规范

**示例**:

- 非功能需求缺少量化指标
- 模块职责描述过于简单
- 风险应对措施不够具体

### INFO (信息) - 可选改进

这些是改进建议，可以提升文档质量：

- 内容组织优化
- 表述方式改进
- 补充说明

**示例**:

- 可以增加流程图
- 术语可以增加解释
- 可以增加参考文档

## 评分标准

| 分数 | 描述 | 评审结论 |
|------|------|----------|
| 90-100 | 优秀：无明显问题，内容完整准确 | ✅ 通过 |
| 75-89 | 良好：有小问题，需针对性改进后方可进入下一阶段 | ⚠️ 条件通过 |
| 0-74 | 不通过：存在明显问题，需要修改后重新评审 | ❌ 不通过 |

**评分计算方法**:

```
基础分 = 100
扣分 = ERROR数量 × 10 + WARNING数量 × 3 + INFO数量 × 1
最终分 = max(0, 基础分 - 扣分)
```

**通过阈值**: 分数 >= 90 直接通过；75-89 条件通过（需改进后确认）；< 75 不通过。

## 评审检查表使用指南

每个阶段的检查表包含该阶段特有的检查项。评审时：

1. **顺序检查**: 按检查表的顺序逐项评估
2. **证据引用**: 指出问题所在的具体位置
3. **建议具体**: 提供可操作的改进建议
4. **客观描述**: 描述问题本身，不要主观评价
