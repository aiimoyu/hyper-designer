# 需求分析检查表

本检查表用于评审需求分析说明书的质量。检查项直接对应模板 `assets/requirements-analysis-template.md` 的结构。

---

## §1 概要

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 名称是否明确 | ERROR | 格式为 `[System Name - Sub-feature Name]`，能区分不同需求 |
| 描述是否清晰 | WARNING | 一句话说明核心目标，"要做什么" |
| 输入来源是否标注 | INFO | 必须追溯来源：`GitHub Issue #N / Product Requirement / Oral` |
| 项目类型是否正确 | WARNING | `Feature Enhancement / New Feature Development` |

---

## §2 项目背景与 5W2H 分析

### 2.1 项目背景

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 需求价值是否具体可追溯 | ERROR | 避免"提升用户体验"等泛泛描述；需追溯到业务根源，如"故障发现时间从15分钟缩短到实时" |
| 需求描述是否界定边界 | ERROR | 说明"要做什么"和本质范围，明确边界 |

### 2.2 5W2H 分析

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 各维度是否完整 | WARNING | What/Why/Who/When/Where/How/How Much 7项齐全 |
| 内容是否具体可验证 | ERROR | 每项 1-2 句，避免模糊表达 |
| Who 是否涵盖所有角色 | WARNING | 主要用户 + 管理员角色，不遗漏 |
| How Much 是否有估算 | INFO | 规模估算、开发资源投入（人力、时间） |

---

## §3 业务功能与场景

### 3.1 功能性需求

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 编号是否规范 | WARNING | 格式为 `FR-001`，按顺序编号 |
| 需求类别是否标注 | WARNING | 标注功能类别 |
| 优先级是否合理 | ERROR | P0/P1/P2 标注清晰；核心主流为 P0，效率辅助为 P1，体验优化为 P2 |
| 需求描述是否完整 | WARNING | 2-3 句总结核心行为、用户价值、包含的子能力 |

### 3.2 非功能性需求

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 编号是否规范 | WARNING | 格式为 `NFR-001` |
| 是否有量化验收标准 | ERROR | 描述包含可量化的验收标准：响应时间、可用率、错误率、恢复时间等 |
| 是否覆盖关键质量属性 | WARNING | 可用性、可靠性、服务性、安全性、性能、可扩展性、兼容性等视角 |

### 3.3 业务规则与约束

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 业务规则是否明确 | WARNING | 校验逻辑、计算公式、状态转换、触发条件、前置条件、互斥规则等 |

### 3.4 主成功场景

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 是否使用具名角色 | ERROR | 推荐"操作员张三"、"审核员李四"描述典型端到端成功路径 |
| 场景流程是否完整 | ERROR | 触发事件 → 操作 → 系统响应 → 用户确认 → 结果生成 → 目标完成 → 后续影响 |

### 3.5 场景列表

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 编号是否规范 | WARNING | 格式为 `S-B01`（业务场景）、`S-O01`（操作场景）、`S-M01`（维护场景） |
| 场景分类是否完整 | WARNING | 业务场景/操作场景/维护场景均已覆盖 |
| 简要说明是否清晰 | WARNING | 触发条件和预期结果已说明 |

---

## §4 性能规格

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 性能指标是否量化 | ERROR | 有具体数值（如 `P95 < 500ms`） |
| 测试条件是否明确 | WARNING | 并发数、数据量等条件已说明 |
| 无基线时是否标注待确认 | ERROR | 如无基线或目标数据，需注明"待负载测试确认"或"待业务方补充"，禁止虚构指标 |

---

## §5 验收方法

### 5.1 验收标准

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 验收项是否可量化测试 | ERROR | 验收标准可量化、可测试、可重现；无法量化时需有明确判断条件 |
| P0 验收项是否完整 | ERROR | P0 验收项阻断发布 |
| 验收方法是否具体 | WARNING | 具体测试方法已说明 |
| 性能验收是否补充百分位 | INFO | P95/P99 百分位指标补充 |

### 5.2 验收流程

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| Mermaid 流程图是否绘制 | WARNING | 单元测试 → 集成测试 → 系统测试 → 验收测试 流程清晰 |
| 各阶段验收内容是否明确 | WARNING | 验收内容、通过标准、负责人已说明 |

### 5.3 测试场景

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 编号是否规范 | WARNING | 格式为 `TC-001` |
| 场景类型是否覆盖 | ERROR | P0 功能至少覆盖正常路径和异常路径；按需补充边界条件 |
| 前置条件是否明确 | WARNING | 初始状态已说明 |
| 操作步骤是否具体 | WARNING | 操作步骤可执行 |
| 预期结果是否明确 | ERROR | 预期结果可验证 |

### 5.4 交付物定义

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 交付物是否列出 | WARNING | 代码实现、测试报告、部署说明、用户手册、监控配置等 |
| 验收标准是否定义 | WARNING | 每个交付物有验收标准 |

---

## §6 约束

### 6.1 技术约束

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 约束类型是否标注 | WARNING | 技术栈、接口协议、数据格式、部署环境、版本依赖等 |
| 影响范围是否说明 | INFO | 若约束只影响特定模块，需说明影响范围 |

### 6.2 合规与安全要求

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 权限控制是否说明 | ERROR | 权限控制描述及验证方法 |
| 审计日志是否说明 | WARNING | 操作审计要求 |
| 数据安全是否说明 | WARNING | 敏感数据、角色隔离、加密传输等 |

---

## §7 附录

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 附录内容是否必要 | INFO | 无特殊需求时可忽略 |
