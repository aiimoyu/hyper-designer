# 需求设计检查表

本检查表用于评审需求设计说明书的质量。检查项直接对应模板 `assets/requirements-design-template.md` 的结构。

---

## §1 概要

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 名称是否与需求分析一致 | ERROR | 必须与需求分析说明书一致 |
| 描述是否清晰 | WARNING | 设计层面一句话描述 |
| 输入来源是否标注 | ERROR | 需求分析说明书路径/版本 |
| 项目类型是否正确 | WARNING | `Feature Enhancement / New Feature / Refactoring` |

---

## §2 设计目标

### 2.1 性能目标

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 指标是否继承自需求分析 | ERROR | 关键操作响应时间等指标与需求分析说明书一致 |
| 测量条件是否明确 | WARNING | 并发条件、数据规模等已说明 |
| 模块分解是否合理 | WARNING | 系统级目标分解到模块/链路节点，有时间分配和分解依据 |
| 假设条件是否说明 | INFO | 无明确数据时标注"待确认"，禁止虚构 |

### 2.2 可用性目标

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 可用性指标是否定义 | INFO | 可用率 ≥ N% |
| RTO/RPO 是否定义 | INFO | 恢复时间目标、数据恢复点目标 |

## §3 架构设计

### 3.1 架构变更总览

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| Mermaid 架构变更图是否绘制 | ERROR | 展示新增🟢/修改🟡/保护🔴/不涉及⚪模块及调用关系 |
| 图例是否标注 | WARNING | 颜色含义：🔵外部用户、🟢新增、🟡修改、🔴保护、⚪不涉及 |
| 接口命名是否规范 | WARNING | 格式 `IF-{Type}{Number}`：E=外部、N=新增内部、M=修改内部、R=复用内部 |

### 3.2 模块变更明细

| 检查项 | 严重程度 | 检查要点 |

## §5 SR-AR 分解与追溯

### 5.1 SR 列表

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| SR 编号是否规范 | WARNING | 格式 `SR-001` |
| SR 数量是否合理 | ERROR | 一般 1-2 个；超过 3 个需用户确认必要性 |
| 对应场景是否标注 | WARNING | 每个 SR 对应需求分析中的主要场景或功能域 |
| 覆盖功能是否说明 | WARNING | F-01, F-02 等功能编号 |

### 5.2 AR 分配

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| AR 编号是否规范 | WARNING | 格式 `AR-001-01`，关联 SR 编号 |
| 每个 SR 的 AR 数量是否合理 | ERROR | 默认 1-2 个，最多 3 个；超过需确认拆分理由 |
| 一 AR 一元素原则 | ERROR | 每个 AR 属于且仅属于一个系统元素（模块/组件/服务） |
| 功能点是否列出 | ERROR | 每个 AR 必须列出具体功能点，禁止模糊描述 |
| 系统元素是否明确 | ERROR | 模块名及操作类型（新增/扩展/依赖） |

### 5.3 依赖矩阵

| 检查项 | 严重程度 | 检查要点 |
|--------|----------|----------|
| 依赖矩阵是否绘制 | WARNING | 模块间关系：扩展/依赖/新增/无关系 |
| 注释是否说明 | INFO | Extension=修改现有、Dependency=调用不修改、New=全新创建 |

### IR→SR→AR 追溯完整性

| IR（需求分析功能） | SR（系统需求） | AR（架构需求） | 追溯状态 |
|--------------------|----------------|----------------|----------|
| FR-001 [功能名] | SR-001 | AR-001-01, AR-001-02 | ✅ 完整 |
| FR-002 [功能名] | SR-001 | AR-001-02 | ✅ 完整 |
| NFR-001 [非功能名] | SR-002 | AR-002-01 | ✅ 完整 |
| ... | ... | ... | □ 逐项检查 |

**最终检查结论**：
- [ ] ✅ IR→SR→AR 追溯链完整，无遗漏（必须）
- [ ] ✅ 无过度设计：SR/AR 可追溯到 IR（警告）
- [ ] ✅ 无需求遗漏：所有 IR 均有实现路径（必须修复）
