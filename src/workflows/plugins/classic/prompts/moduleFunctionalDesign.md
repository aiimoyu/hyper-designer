## 当前阶段：模块功能设计

**阶段标识**: `moduleFunctionalDesign`  
**执行Agent**: HEngineer  
**核心目标**: 为每个模块输出详细的技术规格，包括职责、接口、内部结构、算法、数据结构、测试策略。

### 1. 输入与资料收集

在开始执行前，必须通过 `document-collector` skill 完成以下资料收集。所需的 **4大类资料** 如下：

| 资料类别 | 关键内容 | 必需性 | 用途说明 |
| :--- | :--- | :--- | :--- |
| **系统设计资料**<br>(前阶段输出) | 系统设计文档（system-design.md） | 必需 | 系统架构、模块划分与接口契约 |
| **系统设计资料**<br>(模块参考) | 同类模块设计方案 | 可选 | 设计模式与最佳实践参考 |
| **代码库资料**<br>(接口规范) | API 设计规范、协议标准 | 可选 | 接口契约定义参考 |
| **代码库资料**<br>(开源实现) | 开源同类模块实现 | 可选 | 详细设计参考 |

**收集动作：**

1. 从工作流状态或阶段输出目录读取 system-design.md 文档（必须）。
2. 与需求方或接口拥有者确认接口契约（路径、方法、输入输出示例、错误码、版本兼容性）。
3. 在开源仓库和内部库中搜索参考实现与设计模式，收集有代表性的实现片段和架构决策记录。
4. 收集并整理测试策略参考（单元/集成/契约测试示例、性能基准、mock 数据模板）。

### 2. 执行规范与 Skill 使用

**核心 Skill**: `functional-design`

此 Skill 支持模块级详细设计，包括类设计、算法定义、接口规范与测试策略。

### 3. 阶段交付物

| 文件名 | 路径 | 格式要求 |
| :--- | :--- | :--- |
| **{模块名}设计.md** | `.hyper-designer/moduleFunctionalDesign/{模块名}设计.md` | Markdown，包含模块概述、公开接口、数据模型、主要算法、错误处理、测试策略 |

**内容要点**：
- 模块概述与职责
- 公开接口（API/消息/事件契约）和示例
- 数据模型与重要类型定义
- 主要算法、流程与伪代码
- 错误与异常处理策略
- 部署、性能与运行时约束
- 测试策略与验收标准（单元、集成、契约、性能）

### 4. 质量审查

**审核标准:**

1. **模块职责单一性**: 模块职责是否单一明确，符合高内聚低耦合原则？
2. **接口定义清晰性**: 接口契约是否完整（路径、方法、输入/输出、错误码、示例）？
3. **测试策略完整性**: 是否覆盖单元、集成、契约、性能等多维度测试？
4. **规范性**: 文档结构是否符合 `functional-design` skill 定义的模板？
