## 当前阶段：用例分析

**阶段标识**: `useCaseAnalysis`  
**执行Agent**: HArchitect  
**核心目标**: 将场景库中的功能场景细化为可执行、可验收的用例规格，明确输入/输出与验收标准。

### 1. 资料收集

**必须先完成资料收集，再开始执行。**
使用 `task` 工具委派 HCollector 进行资料收集

#### 所需资料类别

| 资料类别 | 关键内容 | 必需性 | 用途说明 |
| :--- | :--- | :--- | :--- |
| **系统需求分析资料**<br>(前阶段输出) | 场景文档（{功能名}场景.md） | 必需 | 细化主场景、备选场景、异常场景 |
| **系统需求分析资料**<br>(风险参考) | FMEA库 | 可选 | 故障模式、异常处理场景参考 |
| **系统需求分析资料**<br>(功能参考) | 功能库 | 参考用 | 功能规格说明参考 |

### 2. 执行规范与 Skill 使用

**核心 Skill**: `use-case-analysis`

此 Skill 将场景细化为详细的用例规格，定义用例的前置条件、成功保证、触发事件、Actor、主成功场景、扩展场景和DFX属性。

**核心参考资料**: `.hyper-designer/document/systemRequirementAnalysis/manifest.md`，由 HCollector 生成，包含已收集资料的结构化摘要和引用链接。
### 3. 阶段交付物

| 文件名 | 路径 | 格式要求 |
| :--- | :--- | :--- |
| **{功能名}用例.md** | `.hyper-designer/useCaseAnalysis/{功能名}用例.md` | Markdown，包含完整的用例规格（前置条件、触发事件、Actor、主成功场景、扩展/异常场景、验收标准） |

**交付节拍**：每个功能在完成后产出单文件

### 4. 质量审查

**审核标准:**

1. **用例规格完整性**: 前置条件、触发事件、Actor、主成功场景、扩展/异常场景是否完整且无歧义？
2. **输入/输出明确性**: 数据格式、类型、边界条件、错误代码/异常分支是否清晰定义？
3. **验收标准可测试性**: 验收条件是否能以自动化或手工测试判定通过/失败？
4. **规范性**: 文档结构是否符合 `use-case-analysis` skill 定义的模板？
