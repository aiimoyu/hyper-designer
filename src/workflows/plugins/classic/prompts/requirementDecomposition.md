# 当前阶段

**你现在处于：需求分解 (Requirement Decomposition / SR-AR 分解) 阶段**

概述：在本阶段，HEngineer 使用功能细化（functionalRefinement）产生的功能列表作为输入，应用领域驱动设计（DDD）与模块化分解方法，将系统级需求（SR）细化为模块级需求与可执行实现要求（AR），并定义子系统接口与依赖关系。

## 必需技能

- `sr-ar-decomposition` - SR→AR 分解方法论与模板
- `ir-sr-ar-traceability` - 建立并验证 IR→SR→AR 的追溯链

## 所需资料

| 资料类别 | 具体资料 | 用途 | 必需性 |
|---------|---------|------|-------|
| 系统需求分析资料 | 功能库 | 功能列表（分解基础） | 必需（前阶段输出） |
| 系统设计資料 | 现有系统架构 | 如果是重构项目 | 如有则必需 |
| 代码库资料 | 参考项目架构模式 | 架构参考、最佳实践 | 可选 |
| 领域资料 | 技术约束 | 部署环境、性能要求 | 必需 |

## 资料收集执行

1. 读取 functionalRefinement 阶段输出的功能列表（{系统名}功能列表.md），作为分解的权威输入。
2. 询问并获取现有系统架构文档（若为重构/迁移项目），包含部署拓扑、模块清单与第三方依赖。
3. 收集技术约束信息：部署环境（云/本地）、性能/并发目标、数据保留/合规性、硬件限制等。
4. 搜索并参考相似开源/企业项目的架构模式（可选），作为实现参考与最佳实践。

## 阶段输出

- 主输出文件：`sr-ar-decomposition.md`（包含 SR 列表、对应 AR、模块接口定义、依赖矩阵与优先级）

## 使用技能

- `sr-ar-decomposition`
- `ir-sr-ar-traceability`

## 质量审核要求（必须遵守）

**强制审核：** 完成本阶段所有输出文档后，**必须**使用 `HCritic` agent 进行设计审核。

- 调用方式：{{TOOL:delegate_critic_review}}
- 审核未通过时：根据 HCritic 的反馈意见，**必须**修改文档并重新提交审核。
- 审核流程：重复 "生成文档 → HCritic 审核 → 修改" 直到审核通过。
- **禁止跳过审核**：未经 HCritic 审核通过的文档不得进入下一阶段。

## 审查要点

- SR-AR 分解合理性：每个 SR 是否被完整且恰当地映射为一个或多个 AR，且无明显遗漏。
- 模块边界清晰性：模块职责、接口与依赖是否明确，是否遵循低耦合高内聚原则。
- 追溯链完整性：从 IR → SR → AR 的追溯链是否完整、可验证，并记录了来源与决策理由。
