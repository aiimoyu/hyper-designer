### 1. 输入与资料收集

**在开始执行前，必须通过读取资料清单和自主搜集完成资料收集。**

所需的 **资料类别定义** 如下：

| 资料类别 | 关键内容 | 必需性 | 用途说明 |
| :--- | :--- | :--- | :--- |
| **系统需求分析资料**<br>(前阶段输出-IR) | 需求信息（需求信息.md） | 必需 | 理解需求背景、约束与业务目标 |
| **系统需求分析资料**<br>(前阶段输出-SR/AR分解) | SR-AR分解文档 | 必需 | 功能实现需求与模块划分基础 |
| **功能细化资料**<br>(功能列表文档) | SR 映射表、NFR/DFX 汇总表 | 必需 | NFR/DFX 汇总用于填充 §1 设计目标与 §7 非功能实现策略 |
| **系统设计资料**<br>(架构参考) | 系统架构模式、设计原则 | 可选 | 架构设计参考 |
| **代码库资料**<br>(技术栈参考) | 技术选型对比、最佳实践 | 可选 | 技术决策支撑 |
| **代码库资料**<br>(开源参考) | 同类开源系统实现 | 可选 | 架构模式参考 |
**资料收集流程**（详见"单阶段处理流程 Step 2"）：

1. **读取资料清单**：读取项目根目录 `REFERENCE.md` 中相关 Section，解析用户填写的资料信息。
2. **确认完整性**：检查必需资料是否已填写，向用户汇报状态并确认是否需要补充。
3. **搜集与解析**：根据资料清单信息读取本地文件和URL，使用 `explore`/`librarian` 自主搜集补充资料。

**核心参考资料**: 根据上下文搜集已收集的前置阶段交付物、系统设计资料、代码库资料等的结构化摘要和引用链接。

### 2. 执行规范与 Skill 使用

**核心 Skill**: `functional-design`

此 Skill 提供架构模板、模块划分方法、数据建模指南与质量检查清单，支持系统架构设计与技术选型。

**模板遵循要求**：system-design.md 输出文档必须严格按照 `functional-design` Skill 中的 `system-design.md` 模板撰写，包含 §0 至 §11 全部 12 个章节，不得跳过或合并任何章节。功能列表文档中的 NFR/DFX 汇总表应作为 §1（设计目标与约束）和 §7（非功能实现策略）的主要输入。

### 4. 模板章节完整性检查表

在提交 system-design.md 前，必须逐项确认以下 12 个章节均已撰写完整：

- [ ] **§0 概要**：系统名称/版本/作者、输入来源、创建日期
- [ ] **§1 设计目标与约束**：性能/可用性/安全/成本目标，技术/合规/部署约束（来源：功能列表 NFR/DFX 汇总）
- [ ] **§2 总体架构**：C4 架构视图（Context + Container）、分层策略、服务边界、部署模型
- [ ] **§3 技术栈与选型理由**：技术栈清单、选型决策记录（含权衡分析）、风险评估
- [ ] **§4 模块设计概要**：各模块职责说明、核心类设计、关键接口定义
- [ ] **§5 数据模型**：核心实体定义、ER 关系图、一致性策略、索引策略
- [ ] **§6 交互协议与接口规范**：API 设计规范、认证授权策略、错误处理、版本管理
- [ ] **§7 非功能实现策略**：性能（缓存/并发）、安全（认证/加密）、可靠性（容错/降级）、可观测性（来源：功能列表 NFR/DFX 汇总）
- [ ] **§8 部署与运维建议**：部署架构、CI/CD 流程、监控告警、故障恢复
- [ ] **§9 风险与权衡记录**：架构权衡决策记录、技术风险评估、迁移路径
- [ ] **§10 测试策略**：单元测试、集成测试、性能测试、故障注入测试
- [ ] **§11 参考资料与附录**：参考文献、术语表、变更历史
