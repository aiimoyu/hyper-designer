# Skill: Functional List Refinement（功能列表梳理）

## 技能概述

**输入**：用例文档（UC-NNN）及上游文档（IR 约束、功能场景）  
**输出**：`{功能名}功能列表.md`（必选）+ `{功能名}FMEA.md`（高风险功能必选）  
**核心价值**：将用例中的业务意图转化为可实现、可测试、可追溯的功能清单，并评估对现有系统的影响

---

## 执行流程总览

```
[输入] 用例文档 + IR约束 + 功能场景
    │
    ▼
【Step 1】读取与理解用例
    │  提取主流程 / 备选流程 / 异常流程 / DFX属性
    ▼
【Step 2】功能拆解（新增 vs 修改）
    │  识别功能点，控制粒度，标注类型
    ▼
【Step 3】活动图分析（复杂流程必须）
    │  PlantUML 泳道图，覆盖全流程分支
    ▼
【Step 4】规则、约束与影响分析
    │  为每个功能梳理业务规则 / 约束 / 对现有功能的影响
    ▼
【Step 5】NFR/DFX 提取与 SR 映射
    │  从上游文档聚合非功能需求，分配SR编号
    ▼
【Step 6】高风险功能识别 → FMEA 分析
    │  判断是否需要 FMEA，若需要则输出 FMEA 文档
    ▼
[输出] 功能列表.md  +  （可选）FMEA.md
```

---

## Step 1：读取与理解用例

**目标**：全面解析用例文档，建立分析基础。

从每个用例中提取以下内容：

| 提取项 | 说明 |
|--------|------|
| 主流程 | 成功路径，将映射为核心功能 |
| 备选流程 | 可选路径，将映射为分支/可选功能 |
| 异常流程 | 失败处理，将映射为容错/降级功能 |
| 业务规则 | 将转化为功能规则约束 |
| DFX 属性 | 性能、安全、可靠、隐私指标，进入 NFR 汇总 |

> **检查点**：若用例缺少 DFX 属性定义，使用微确认问题向用户确认，不得自行假设。

---

## Step 2：功能拆解

**目标**：将用例流程拆解为粒度合适的功能点，明确每个功能是「新增」还是「修改」。

### 粒度控制原则

```
❌ 太粗：整个模块作为一个功能（如「用户管理」）
✅ 合适：单一职责、独立可实现、独立可测试（如「用户登录认证」）
❌ 太细：每个字段校验都是一个功能
```

### 用例到功能的映射规则

```
用例主流程   → 核心功能（Must 优先）
用例备选流程 → 可选/分支功能（Should / Could）
用例异常流程 → 容错/降级功能（Must / Should）
```

### 新增 vs 修改的判断标准

- **新增功能**：现有系统中完全不存在对应实现，需要全新开发
- **修改功能**：现有功能需调整行为、规则或数据结构

### 工作草稿格式

```markdown
## 用例到功能映射草稿

- UC-001 用户登录 → F001（新增：登录认证）、F002（新增：登录失败处理）
- UC-002 权限管理 → F003（新增：角色配置）、F010（修改：用户管理，增加角色字段）

## 初步优先级（MoSCoW）
- Must: F001, F002
- Should: F003
- Could: F004

## 待确认问题
- [ ] F003 职责边界是否包含角色继承逻辑？
```

完成草稿后，使用**功能确认对话**与用户同步（见「与用户协作模式」章节）。

---

## Step 3：活动图分析（复杂流程必须）

**触发条件**（满足任一即需绘制）：
- 流程包含 3 个以上判断分支
- 涉及多个角色/系统交互
- 存在多条异常处理路径
- 流程中有并发或等待节点

**绘制原则**：
- 使用 PlantUML 语法
- 多角色场景必须使用泳道（swimlane）
- 必须覆盖：主流程 + 所有备选流程 + 所有异常流程
- 活动图嵌入功能列表文档的「功能分析思路」章节

### 基础活动图模板

```plantuml
@startuml {功能名称}_活动图
skinparam activity {
  BackgroundColor #F5F5F5
  BorderColor #333333
  FontSize 12
}

start
:步骤1 描述;

if (判断条件?) then (是)
  :分支A操作;
else (否)
  :分支B操作;
endif

stop
@enduml
```

### 泳道活动图模板（多角色）

```plantuml
@startuml {功能名称}_泳道活动图

|用户|
start
:用户发起操作;

|系统|
:系统校验请求;

if (校验通过?) then (是)
  |数据库|
  :持久化数据;
  |系统|
  :返回成功响应;
else (否)
  :返回错误信息;
endif

|用户|
:查看操作结果;
stop

@enduml
```

## Step 4：规则、约束与影响分析

**目标**：为每个功能补充业务细节，并分析对现有功能的影响。

### 每个功能需填写的内容

| 字段 | 说明 | 示例 |
|------|------|------|
| 功能内容 | 1-2 句话描述功能做什么 | 验证用户身份，创建登录会话 |
| 功能规则 | 业务规则，来自用例规格 | 密码 8-20 位，含字母和数字 |
| 功能约束 | 技术/业务边界 | 会话有效期 24 小时 |
| 影响分析 | 对现有功能的增/删/改影响 | F010 用户管理（改）：增加锁定状态字段 |
| 优先级 | MoSCoW | Must |

### 影响类型说明

- **增**：现有功能需新增字段、接口或数据表
- **改**：现有功能的行为、逻辑或数据结构需要调整
- **删**：现有功能的某些能力需要移除

> 发现影响现有功能时，主动通知用户（见「影响分析确认」）。

## Step 5：NFR/DFX 提取与 SR 映射

### NFR/DFX 提取

**来源**（必须全部扫描）：

| 来源 | 文档 | 标注方式 |
|------|------|---------|
| IR 约束 | 需求信息.md | 标注为 `IR 约束` |
| 场景需求 | 功能场景.md | 标注为 `场景需求` |
| 用例 DFX | 各用例文档的 DFX 属性章节 | 标注为 `UC-NNN DFX` |

**DFX 分类**：

| 类别 | 关注指标 |
|------|---------|
| 性能 | 响应时间、吞吐量、并发数 |
| 安全 | 认证等级、加密标准、权限控制 |
| 可靠 | 可用性、故障恢复、数据完整性 |
| 隐私 | 数据脱敏、访问审计、合规性 |

**整合规则（strictest-metric）**：

- 数值型指标：取最严格值（响应时间取最小，可用性取最大）
- 枚举型指标：取最高标准（AES-256 优于 AES-128）
- 冲突时优先级：`IR 约束 > 场景需求 > UC-NNN DFX`

### SR 映射

**格式**：`SR-NNN`（三位数字，从 001 递增）

**可追溯链**：
```
UC-NNN（用例）→ F-NNN（功能）→ SR-NNN（系统需求）
```

**可追溯性检查**：
- 每个功能至少对应一个 SR
- 每个 SR 描述独立完整，可被下游分解为 AR
- SR 编号不重复、不遗漏

## Step 6：高风险功能识别与 FMEA 分析

### 高风险功能识别标准

满足以下任一条件，该功能需进行 FMEA 分析：

| 风险类型 | 典型场景 |
|---------|---------|
| 业务关键路径 | 支付、订单提交、数据迁移 |
| 不可逆操作 | 数据删除、账号注销、资金转移 |
| 外部依赖 | 第三方 API、硬件接口 |
| 并发敏感 | 库存扣减、余额变更 |
| 安全敏感 | 认证、授权、敏感数据处理 |

### FMEA 分析执行步骤

**步骤 1**：列举该功能所有可能的失效模式（"什么会出错？"）

**步骤 2**：分析每种失效模式的影响后果（"出错后会怎样？"）

**步骤 3**：评估严重度（S）和发生概率（O）

**步骤 4**：计算风险等级 = S × O

```
低风险：S × O ≤ 6
中风险：7 ≤ S × O ≤ 15
高风险：S × O ≥ 16
```

**步骤 5**：针对中/高风险条目提出具体建议措施

### 严重度（S）评级

| 等级 | 描述 |
|-----|------|
| 1 | 影响极小，用户几乎察觉不到 |
| 2 | 轻微影响，用户体验稍差 |
| 3 | 中等影响，功能部分降级 |
| 4 | 严重影响，核心功能失效 |
| 5 | 灾难性影响，数据丢失或资金损失 |

### 发生概率（O）评级

| 等级 | 描述 |
|-----|------|
| 1 | 极少发生（< 0.1%） |
| 2 | 偶尔发生（0.1% - 1%） |
| 3 | 有时发生（1% - 5%） |
| 4 | 经常发生（5% - 20%） |
| 5 | 频繁发生（> 20%） |

### FMEA 输出格式

**输出文件**：`{功能名}FMEA.md`

| 编号 | 功能需求 | 失效模式 | 影响后果 | 严重度 S | 发生概率 O | 现有控制措施 | 风险等级 | 建议措施 | 负责人 |
|-----|---------|---------|---------|---------|----------|------------|---------|---------|-------|
| FMEA-001 | [功能] | [失效描述] | [后果描述] | [1-5] | [1-5] | [现有措施] | 低/中/高 | [建议] | [负责人] |

## 与用户协作模式

### 功能确认对话（Step 2 完成后触发）

```
"我从用例中提取了 X 个功能点，请确认：

【新增功能】
- F001 用户登录认证（Must）：验证用户身份，创建会话
- F002 登录失败处理（Must）：记录失败次数，超限锁定账号

【修改功能】
- F010 用户管理（Should）：需增加账号锁定状态字段

确认项：
1. 功能拆分粒度是否合理？
2. 是否有遗漏的功能？
3. 优先级（MoSCoW）是否准确？"
```

### 影响分析确认（发现现有功能影响时触发）

```
"新功能「登录失败处理」会影响现有功能：
- F010 用户管理（改）：需增加账号锁定状态字段
- F011 日志记录（改）：需新增登录失败事件类型

请确认：是否需要评估版本兼容性或调整现有功能设计？"
```

### 微确认问题（发现信息缺失时触发）

```
"用例 UC-NNN 的「性能」DFX 属性未提供量化指标。
请提供具体数值（如响应时间 < 200ms），或确认暂不设定。"
```

```
"功能 F-NNN 与 F-MMM 的职责存在重叠：[具体描述]。
请确认拆分方式，或说明归属模块。"
```

```
"以下功能未找到对应用例：[功能列表]。
是否需要回退到用例分析阶段补充？"
```

## 输出文件规范

### 主输出：`{功能名}功能列表.md`

参见 `references/functional-list-template.md` 中的完整模板格式。

文档结构：
1. 功能分析思路与结果总结（含活动图）
2. SR 映射表
3. 功能描述（每个功能：内容 / 规则 / 约束 / 影响分析 / 优先级）
4. NFR/DFX 汇总表

### 辅助输出：`{功能名}FMEA.md`（仅高风险功能）

格式参见本文档 Step 6 的 FMEA 输出格式。

## 质量检查清单

在提交文档前，逐项核对：

**功能完整性**
- [ ] 所有用例均已映射到功能，新增/修改标注清晰
- [ ] 每个功能描述为 1-2 句话，职责单一
- [ ] 每个功能的业务规则已整理
- [ ] 每个功能的约束条件已列出

**流程分析**
- [ ] 复杂用例（3个以上分支）已绘制活动图
- [ ] 活动图覆盖主流程、备选流程、异常流程

**影响与追溯**
- [ ] 每个功能的影响分析表格已填写
- [ ] 每个功能已分配 SR 编号，SR 映射表完整
- [ ] 可追溯链 UC→F→SR 无断点

**非功能需求**
- [ ] NFR/DFX 已从所有上游文档（IR/场景/用例）提取
- [ ] 同类别指标已按 strictest-metric 规则整合
- [ ] 每条 NFR 已标注来源和受影响的功能编号

**优先级与风险**
- [ ] 所有功能已标注 MoSCoW 优先级
- [ ] 高风险功能已识别并完成 FMEA 分析

## 成功标准

| 标准 | 验证方式 |
|------|---------|
| 功能列表完整，覆盖所有用例 | 用例数 vs 功能映射数 |
| 每个功能有明确的业务规则和约束 | 逐条检查功能描述 |
| 影响分析准确，可指导现有功能调整 | 与用户确认影响范围 |
| MoSCoW 优先级合理 | 用户确认 |
| SR 映射完整，支持下游 AR 分解 | 可追溯链无断点 |
| NFR/DFX 汇总完整 | 覆盖所有 DFX 类别 |
| 高风险功能完成 FMEA 分析 | FMEA 文档存在且条目完整 |

**准备进入下一阶段的信号**：
- 用户已确认功能列表
- 无阻塞性业务问题
- 所有质量检查项均已通过
