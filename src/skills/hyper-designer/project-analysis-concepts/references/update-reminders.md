# Update Reminders

All generated analysis artifacts must include self-reference declarations to enable maintenance when code changes.

## Purpose

Update reminders serve two goals:

1. **Self-Reference**: Each document declares "when I change, update me"
2. **Maintenance Trigger**: Clear signals for when documentation needs updates

## Reminder Format

### Standard Reminder Block

Place at the end of each generated markdown file:

```markdown
---

## Update Reminder

This document was generated by project-analysis workflow on **[Date]**.

### When to Update

Update this document when:

- [ ] Source code structure changes (new files, moved files, deleted files)
- [ ] Dependencies change (new libraries, version updates)
- [ ] APIs change (endpoints added/modified/removed)
- [ ] Data models change (schema modifications)
- [ ] Architecture patterns change (new patterns, refactoring)
- [ ] Business logic changes (rules, validations, flows)

### How to Update

1. Re-run project-analysis workflow: `hyper-designer workflow projectAnalysis`
2. Or manually update affected sections based on code changes
3. Verify all code citations still reference valid files
4. Validate all Mermaid diagrams render correctly
5. Check coverage report for new gaps

### Last Analysis

- **Date**: [ISO-8601 timestamp]
- **Workflow Version**: [version]
- **Analyzed By**: HAnalysis agent
- **Target Path**: [target project path]

### Related Artifacts

- System Analysis: [architecture.md](../architecture.md)
- Component Manifest: [_meta/component-manifest.json](_meta/component-manifest.json)
- Coverage Report: [coverage-report.md](../coverage-report.md)
```

### Simplified Reminder (for short sections)

```markdown
> **Update Reminder**: This section reflects code as of [Date]. When [specific code area] changes, update this section to maintain accuracy.
```

## Trigger Conditions

Define specific triggers for each artifact type:

### architecture.md Triggers

```markdown
Update this document when:

- [ ] **Structure changes**: New modules, reorganized directories, renamed files
- [ ] **Dependency changes**: New imports, removed dependencies, version updates
- [ ] **Architecture changes**: New layers, pattern shifts, refactoring
- [ ] **Data flow changes**: New endpoints, modified request/response flows
- [ ] **State changes**: New state variables, modified state management
- [ ] **Pattern changes**: New design patterns, identified anti-patterns
```

### component/{componentSlug}.md Triggers

```markdown
Update this document when:

- [ ] **Responsibility changes**: Component scope expands or contracts
- [ ] **Interface changes**: New public methods, modified signatures
- [ ] **Data changes**: New data structures, modified state
- [ ] **Position changes**: Layer reassignment, subdomain changes
- [ ] **Dependency changes**: New component dependencies, removed references
```

### coverage-report.md Triggers

```markdown
Update this document when:

- [ ] **Any code changes**: New files, modified files, deleted files
- [ ] **Analysis updates**: architecture.md or component/*.md changes
- [ ] **Manifest updates**: component-manifest.json or source-inventory.json changes
```

## Update Workflow

### Automated Update (Recommended)

When code changes, re-run the entire workflow:

```bash
# From target project directory
hyper-designer workflow projectAnalysis
```

The workflow will:
1. Detect existing manifests in `_meta/`
2. Compare timestamps with source files
3. Only regenerate changed artifacts
4. Update all reminders with new timestamps
5. Validate coverage

### Manual Update (For Small Changes)

For isolated changes, manually update affected sections:

1. Identify changed code areas
2. Locate corresponding documentation sections
3. Update citations and descriptions
4. Update Mermaid diagrams if needed
5. Update reminder timestamp
6. Validate all citations still resolve

### Incremental Update Strategy

Use timestamps to minimize regeneration:

```json
// In manifest.json
{
  "updatedAt": "2026-03-12T10:30:00Z",
  "lastUpdate": {
    "architecture.md": "2026-03-12T10:30:00Z",
    "component/auth-service.md": "2026-03-12T10:30:00Z",
    "component/user-repository.md": "2026-03-12T10:30:00Z"
  }
}
```

When re-running:
- Compare file modification times with `lastUpdate` timestamps
- Only regenerate artifacts older than source files
- Update `lastUpdate` for regenerated artifacts

## Validation

Stage 3 validates update reminders:

### Reminder Presence Check

- [ ] Every markdown file has an update reminder section
- [ ] Reminder includes date and workflow version
- [ ] Reminder lists trigger conditions
- [ ] Reminder references related artifacts

### Timestamp Consistency Check

- [ ] Reminder date matches manifest `updatedAt` timestamp
- [ ] All artifact timestamps are consistent
- [ ] No future timestamps detected

### Citation Validity Check

- [ ] All code citations reference existing files
- [ ] All line ranges are within file bounds
- [ ] No orphan citations (files deleted but still cited)

## Best Practices

### DO ✅

- Include update reminders in every generated artifact
- Use standard reminder format for consistency
- Specify clear trigger conditions
- Include related artifact links
- Update timestamps when regenerating
- Validate citations after manual updates

### DON'T ❌

- Omit update reminders
- Use inconsistent reminder formats
- Skip trigger condition lists
- Forget to update timestamps
- Leave orphan citations after deletions
- Assume documentation stays accurate

## Automation Hooks

Update reminders can be integrated with CI/CD:

### Pre-Commit Hook

```bash
#!/bin/bash
# Check if code changes affect documented areas

CHANGED_FILES=$(git diff --name-only HEAD)

if echo "$CHANGED_FILES" | grep -q "src/"; then
    echo "⚠️  Code changes detected. Consider re-running project-analysis workflow."
    echo "   Run: hyper-designer workflow projectAnalysis"
fi
```

### Post-Merge Hook

```bash
#!/bin/bash
# Auto-regenerate analysis after merges

echo "🔄 Running project-analysis workflow..."
hyper-designer workflow projectAnalysis
```

## Maintenance Schedule

### Regular Maintenance

- **Weekly**: Check for new code not yet documented
- **Monthly**: Full workflow re-run to ensure coverage
- **After major changes**: Immediate re-run

### Coverage Monitoring

Monitor coverage-report.json trends:

```json
{
  "verdict": "pass",
  "summary": {
    "coveragePercentage": 95.2
  },
  "trends": {
    "lastWeek": 94.8,
    "lastMonth": 92.1
  }
}
```

If coverage drops below threshold, trigger update:
- Below 90%: Warning
- Below 80%: Alert
- Below 70%: Critical - immediate re-run required
