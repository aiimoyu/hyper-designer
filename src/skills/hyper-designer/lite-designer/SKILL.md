# Lite Designer

三步轻量化需求设计流程，适用于功能增强型需求或中小规模新功能开发。

```
需求场景分析 → 功能与模块设计 → SDD 开发计划
     ↓                ↓               ↓
需求分析说明书    需求设计说明书      开发计划
```

**不适用场景**：全新系统从零设计、需要深度领域建模、多团队协作的大型项目（建议使用完整 hyper-designer 流程）。

---

## 启动流程：材料收集 + 场景确认

**每次启动流程，先完成这两步，再进入任何 Phase。**

### Step 0-A：一次性收集所有材料

把所有后续阶段可能需要的输入**一次性向用户收集**，避免在设计阶段反复打断。根据需求类型按需选择提问内容：

```
在开始之前，请按情况补充以下材料（没有的填"无"）：

1. 代码库
   - 本项目代码路径 / 仓库链接 / 粘贴关键模块结构：
   - 参考项目代码（如有类似实现可参考）：

2. 领域资料（如有）
   - 领域架构分析 / 领域威胁分析 / 合规要求 / 特殊领域需求：

3. 现有系统资料（如有）
   - 场景库 / FMEA 库 / 功能库：

4. 设计参考（如有）
   - 业界设计参考 / 现有系统设计说明书 / 模块功能设计说明书：
```

**简化规则**：
- 全新功能且无现有系统 → 只询问领域资料
- 用户说"直接开始" → Phase 1 先不阻塞，在 Phase 2 开始前再索取代码库
- **Phase 2 设计阶段必须有代码库，或明确确认"无现有代码库"**

### Step 0-B：场景确认（输入模糊时必做）

分析输入后、输出正式文档前，**先用自然语言描述场景，请用户确认**：

```
在整理文档前，确认一下我的理解：

**典型场景**：[2-4 句话，描述：谁 → 做什么 → 系统怎么响应 → 结果是什么]

这个场景理解正确吗？有遗漏的关键场景吗？
如有需要调整，请告知，我再统一修改后输出文档。
```

如果输入已经足够清晰（如完整 Issue），可简化为一句话确认后直接输出。

## 三步流程总览

### Phase 1 — 需求场景分析
**目标**：理解需求本质，识别功能边界，产出结构化的需求分析说明书。

**⚠️ 强制要求**：执行本阶段前，**必须先读取** `references/phase1-requirements-analysis.md`，该文件包含完整的输出模板结构、各章节写作指南和质量自检清单。**严禁跳过此步骤，严禁使用简化模板。**

**步骤**：Step 0-A 材料收集 → Step 0-B 场景确认 → 5W2H 分析 → 功能清单 → 验收 & 约束 & 边界

**交互节点**：输出初稿后暂停，确认用户修改意见，再继续 Phase 2。

**输出**：`需求分析说明书.md`

---

### Phase 2 — 功能与模块设计
**目标**：将需求分解为可实现的系统设计，产出带 SR-AR 追溯的需求设计说明书。

**⚠️ 强制要求**：执行本阶段前，**必须先读取** `references/phase2-functional-module-design.md`，该文件包含完整的 SR-AR 分解原则、输出模板结构、接口定义规范和修改围栏确认流程。**严禁跳过此步骤，严禁使用简化模板。**

**步骤**：确认代码库 → 设计目标 → SR 分解 → AR 分配（含功能点） → 模块接口 → 依赖矩阵 → 修改围栏（历史文档+代码库分析+用户确认）

**SR 分解原则**：SR 总量控制在 2-6 个；每个 SR 对应一个主要场景；新增 SR 前先自问是否可合并；每个 SR 下包含多个 AR。

**AR 分配原则**：每个 SR 默认对应 1-2 个 AR，最多不超过 3 个；超过 3 个 AR 需与用户确认；AR 应归属不同的系统元素或有明确的技术栈差异。

**围栏确认流程**：修改围栏确定前必须：1) 分析历史文档中的 Out of Scope 和模块边界；2) 使用 SearchCodebase/Grep 分析代码库模块结构；3) 使用 AskUserQuestion 与用户确认围栏范围。

**交互节点**：先展示 SR-AR 结构草稿请用户确认，再补全细节。

**输出**：`需求设计说明书.md`

---

### Phase 3 — SDD 开发计划
**目标**：基于设计说明书，生成结构化开发任务计划，可直接交给开发 Agent 执行。

**⚠️ 强制要求**：执行本阶段前，**必须先读取** `references/phase3-sdd-plan.md`，该文件包含完整的任务分解模板、验收条件格式和测试计划要求。**严禁跳过此步骤，严禁使用简化模板。**

**步骤**：AR → 开发任务 → 里程碑排序 → 依赖关系 → 验收条件 → 执行说明

**交互节点**：输出前确认里程碑时间节点是否符合预期。

**输出**：`开发计划.md`

## 对话原则

### 批量提问，减少打断
- 每次**至多问 4 个问题，一次全问**，不分轮追问
- 优先用"场景描述 + 是否正确"的方式确认，而非开放追问
- 多个不确定点时，优先选最影响方向的 1-2 个先问

### 按需交互
| 情况 | 交互方式 |
|------|----------|
| 输入充分（完整 Issue/PRD） | 直接出文档，输出后征求修改意见 |
| 输入模糊（口述/一句话） | Step 0-B 场景确认后再出文档 |
| Phase 2 缺代码库 | 明确列出缺失项，一次性请用户补充 |
| SR-AR 有歧义 | 展示草稿结构让用户确认，再补细节 |

---

## 文档质量门（输出前自检）

**Phase 1**：每条功能有优先级 / 主成功场景是具体操作流 / Out of Scope 已列关键排除项

**Phase 2**：每个 SR 对应主要场景 / 新增 SR 已自问是否可合并 / 每个 AR 列出具体功能点 / 追溯链完整 / 修改围栏已标注

**Phase 3**：每个 AR 对应开发任务 / 每个任务有可客观判断的验收条件 / 任务依赖关系已标注
