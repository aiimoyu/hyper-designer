# Phase 1 — 需求场景分析

## 目标

将原始需求输入（GitHub Issue、口述、功能请求等）转化为结构化的**需求分析说明书**，明确功能边界、优先级和验收标准。

---

## 输入处理策略

### 输入充分时（推荐路径）

当用户提供了完整的需求描述时，**跳过多轮提问，直接分析并给出场景确认**：

```
我理解这个需求的核心场景是：

**典型场景**：
[用 2-4 句话描述：谁 / 在什么情况下 / 做了什么 / 系统如何响应 / 最终结果]

这个理解正确吗？有需要补充或调整的场景吗？确认后我将输出正式文档。
```

### 输入模糊时（批量提问）

若需求描述过于简略，**一次性提出所有关键问题**（不超过 4 个），等用户回答后再进行场景确认：

```
为了准确分析需求，我有几个关键问题：

1. 这个功能解决了什么具体痛点？现有方式有什么不足？
2. 主要使用者是谁？他们最典型的操作流程是什么？
3. 哪些功能是绝对不能少的（P0），哪些是锦上添花？
4. 有没有明确的技术约束、不能碰的边界，或已知的参考实现？
```

收到回答后，立即进行场景确认，再输出文档，**不再追加新一轮提问**。

## 各节写作指南

### §0 概要

| 信息 | 内容 |
|------|------|
| **名称** | [系统名称 - 子功能名称] |
| **描述** | [一句话说明要做什么] |
| **输入来源** | [GitHub Issue #N / 产品需求 / 口述] |
| **项目类型** | 功能增强型需求 / 新功能开发 |

---

### 项目背景

**需求价值**（1-2 句）：该需求为业务带来的核心价值，说明"为什么值得做"。追溯到业务根源，避免泛泛而谈（❌"提升用户体验"→ ✅"将故障发现时间从 15 分钟缩短至实时"）。

**需求描述**（1-2 句）：需求的本质和范围，说明"要做什么"。

---

### 5W2H 分析

每项限 1-2 句，聚焦核心信息，**避免抽象描述**：

| 维度 | 说明要点 | 常见错误 |
|------|---------|---------|
| **What** | 核心功能内容，具体可验证 | ❌"支持用户交互" → ✅"在故障模式抽取节点增加用户确认/修正入口" |
| **Why** | 追溯到业务根源和痛点 | ❌"提高效率" → ✅"现有自动生成准确率不足，需人工干预提升质量" |
| **Who** | 主要使用者 + 管理者，涵盖所有关键角色 | 不要遗漏管理员角色 |
| **When** | 使用时机和触发条件 | |
| **Where** | 物理环境、技术平台、系统上下文 | |
| **How** | 用户核心操作流程 | |
| **How Much** | 规模预估、开发资源投入（人月） | 需包含人力和时间 |

---

### §3 业务功能与场景

#### 3.1 功能清单与优先级

| 功能编号 | 功能名称 | 功能描述 | 优先级 |
|----------|----------|----------|--------|
| F-01 | [名称] | [2-3 句描述核心行为和用户价值，说明包含哪些子能力] | **P0 - Must Have** |
| F-02 | [名称] | [描述] | **P1 - Should Have** |
| F-03 | [名称] | [描述] | **P2 - Nice to Have** |

**优先级定义**：
- **P0 Must Have**：缺少则需求无法完成，必须实现
- **P1 Should Have**：核心增值功能，计划实现
- **P2 Nice to Have**：锦上添花，有资源再做

#### 3.2 主成功场景

用具名角色（如"运维人员张三"）描述最典型的端到端操作流：

```
[角色] 遇到 [触发事件]
→ [角色] 执行 [操作A]
→ 系统 [响应B]
→ [角色] 查看 / 确认 / 修正
→ 系统 [根据输入生成结果C]
→ [角色] 完成目标
→ 系统记录 [后续影响/知识沉淀]
```

#### 3.3 业务场景分类

| 类型 | 定义 | 本需求中的例子 |
|------|------|--------------|
| **业务场景** | 直接产生业务价值的核心操作 | [举例] |
| **操作场景** | 日常使用的辅助操作（搜索、筛选、导出） | [举例] |
| **维护场景** | 管理员配置和运维操作 | [举例] |
| **制造场景** | 系统批量生成或处理内容 | [举例] |

### §6 验收方法

#### 6.1 验收标准与方法

| 验收项 | 验收标准 | 验收方法 | 优先级 |
|--------|----------|----------|--------|
| [项目名] | [可量化标准，含基线说明。如："经反馈修正后准确率较基线提升 ≥20%；基线=现有无反馈机制的结果"] | [具体测试方法] | P0 |

**写作要点**：
- 标准必须**可量化**（有数字、有基线、有测试方法）
- P0 验收项阻断发布，P1/P2 可延期

#### 6.2 验收流程

```
单元测试 → 集成测试 → 系统测试 → 验收测试（真实场景验证）
```

#### 6.3 交付物定义

| 交付物 | 描述 |
|--------|------|
| 需求分析说明书 | 本文档 |
| 需求设计说明书 | Phase 2 输出 |
| 代码实现 | 功能代码 + 单元测试 |
| 测试报告 | 功能测试 + 性能测试 |
| 用户手册 | 操作说明（如有） |

### §7 边界

#### 7.1 做什么（In Scope）

| 范围 | 描述 |
|------|------|
| [功能域 1] | [具体范围描述] |
| [功能域 2] | [具体范围描述] |

#### 7.2 不做什么（Out of Scope）

| 范围 | 描述 | 不做原因 |
|------|------|----------|
| [排除项 1] | [描述] | [超出范围 / 另立需求 / 技术限制] |
| [排除项 2] | [描述] | [原因] |

> **提示**：Out of Scope 要显式标注那些"看起来相关但本次不做"的事情，这是防止范围蔓延的关键。

## Phase 1 完成后的交互节点

输出需求分析说明书初稿后，**必须暂停并询问**：

```
以上是需求分析说明书初稿。

请确认：
1. 主成功场景的描述是否准确？
2. 功能优先级（P0/P1/P2）是否合理？
3. Out of Scope 有没有遗漏或误放的内容？

确认后进入 Phase 2 功能与模块设计。
```

---

## 质量自检

- [ ] 功能清单中每条都有优先级（P0/P1/P2）
- [ ] 主成功场景是具体操作流，不是抽象描述
- [ ] 验收标准有量化指标和测试方法
- [ ] 技术约束包含性能数值
- [ ] Out of Scope 列出了关键排除项
- [ ] 所有内容基于用户提供的信息，无无据假设
