# IR 5W2H 输出模板

该文件定义了 IR（初始需求）分析的标准输出格式。

## 输出文件

`ir信息.md` - 存放于工作目录根目录

## 文件结构

```markdown
# IR信息 - [系统/产品名称]

## 一句话总结
基于分析形成IR的一句话总结，简明扼要地描述系统核心目标和价值。

**示例**：
- 为投资部门构建一个实时风险监控平台，能够在市场波动时自动预警并提供决策支持。
- 面向下游产品的用户行为数据分析服务，支持实时查询和可视化报表生成。

---

## 5W2H 结构化分析

### Who - 利益相关者
**定义**：需求的利益相关者，系统上下文中的人或物等主语

**内容要求**：
- 列出所有直接和间接的利益相关者
- 区分主要用户和次要用户
- 包括系统、组织、法规等"非人"主体

**示例**：
```
- 主要用户：投资部门交易员（约50人）
- 次要用户：风控部门分析师（约10人）
- 系统主体：下游风控系统、合规审计系统
- 外部主体：金融监管机构、法律法规
```

### What - 需求诉求
**定义**：站在利益相关者角度，对整个系统的期望诉求

**内容要求**：
- 具体化描述，不能过于抽象
- 同一个IR内的诉求必须有强相关性
- 不能在一个IR内写多个不相关的诉求
- 使用"系统应该..."的格式

**示例**：
```
- 系统应该实时监控投资组合的风险敞口
- 系统应该在风险指标超过阈值时自动发送预警
- 系统应该提供历史风险数据的可视化分析
```

**反例**（过于抽象）：
```
- 系统应该让用户满意
- 系统应该性能很好
```

### When - 使用时机
**定义**：需求在哪个生命周期使用

**内容要求**：
- 明确使用场景的时间特征
- 区分实时/准实时/离线场景
- 说明使用频率（持续/周期性/一次性）

**示例**：
```
- 实时监控：交易日 09:00-15:00 持续运行
- 日终处理：每个交易日收盘后 15:30-16:00
- 月度报告：每月第一个工作日生成
- 应急响应：风险事件触发时立即启动
```

### Why - 需求原因
**定义**：需求产生的原因，为什么会提出这个需求

**内容要求**：
- 追溯需求的业务根源
- 说明当前痛点或问题
- 解释需求的价值和必要性

**示例**：
```
- 当前风险监控依赖人工，存在15分钟延迟，无法满足实时风控要求
- 监管新规要求投资合规报告必须在T+1日提交，现有流程无法满足
- 历史上曾出现因风险发现延迟导致200万损失的案例
```

### Where - 使用环境
**定义**：需求产生的环境，需求提出方在什么环境中使用该需求

**内容要求**：
- 物理环境（办公地点、设备条件）
- 技术环境（网络、系统、平台）
- 业务环境（组织架构、流程上下文）

**示例**：
```
- 物理环境：总部交易大厅（50台工作站）、分支机构（VPN接入）
- 技术环境：内网环境，禁止外网访问，需通过统一认证平台登录
- 业务环境：需在现有风控平台基础上集成，对接核心交易系统
```

### How Much - 规格约束
**定义**：需求提出方对当前需求的规格约束

**内容要求**：
- 性能指标（响应时间、吞吐量、并发数）
- 容量指标（数据量、存储、用户数）
- 质量指标（可用性、准确性、覆盖率）
- 资源约束（预算、人力、时间）

**示例**：
```
- 性能：风险计算延迟 < 1秒（P99）
- 并发：支持 100 交易员同时在线
- 容量：存储 5年 历史数据（约10TB）
- 可用性：99.95%，年停机时间 < 4.5小时
- 预算：开发成本 < 200万，年度运维 < 50万
```

### How - 使用方式
**定义**：需求提出方怎么使用该需求

**内容要求**：
- 用户的操作流程
- 系统的交互方式
- 与其他系统的集成方式

**示例**：
```
- 用户通过Web界面登录系统，查看实时风险仪表盘
- 系统通过API推送预警信息到企业微信
- 风控人员可导出PDF格式的风险报告
- 支持通过SQL查询历史数据进行自定义分析
```

## 质量检查清单

在生成 ir信息.md 前，逐一检查：

- [ ] 一句话总结是否清晰传达了核心价值？
- [ ] Who 是否涵盖了所有关键利益相关者？
- [ ] What 是否具体、可验证，而非抽象描述？
- [ ] 同一个 IR 内的诉求是否具有强相关性？
- [ ] When 是否明确了时间和频率特征？
- [ ] Why 是否追溯到了业务根源和痛点？
- [ ] Where 是否覆盖了物理、技术、业务环境？
- [ ] How Much 是否包含可衡量的规格约束？
- [ ] How 是否描述了具体的使用方式和流程？
- [ ] 所有内容是否基于与用户的对话和上下文信息？

## 输出示例

```markdown
# IR信息 - 实时投资风险监控系统

## 一句话总结
为投资部门构建一个实时风险监控平台，能够在市场波动时自动预警并提供决策支持，将风险发现时间从15分钟缩短至1秒以内。

## 5W2H 结构化分析

### Who - 利益相关者
1. **主要用户**：
   - 投资部门交易员（约50人）- 日常使用实时监控功能
   - 风控部门分析师（约10人）- 进行风险分析和报告

2. **系统/组织主体**：
   - 下游风控系统 - 接收风险数据用于综合分析
   - 核心交易系统 - 提供实时交易数据
   - 合规审计系统 - 存储合规检查所需的风险日志

3. **外部主体**：
   - 证监会等金融监管机构 - 接收合规报告
   - 公司内部审计部门 - 定期审查风险管控情况

### What - 需求诉求
1. **FR-001**：系统应该实时监控投资组合的风险敞口，包括市场风险、信用风险、流动性风险
2. **FR-002**：系统应该在风险指标超过预设阈值时，在1秒内自动发送多渠道预警（界面弹窗+企业微信+邮件）
3. **FR-003**：系统应该提供历史风险数据的多维度可视化分析，支持自定义时间范围和指标组合
4. **FR-004**：系统应该支持风险情景模拟和压力测试，允许交易员评估潜在风险影响

### When - 使用时机
1. **实时持续监控**：
   - 时间：交易日 09:00-15:00 连续运行
   - 频率：每秒钟刷新一次风险指标
   
2. **日终处理**：
   - 时间：每个交易日收盘后 15:30-16:00
   - 内容：生成日终风险报告，归档当日数据
   
3. **应急响应**：
   - 触发条件：风险指标超过紧急阈值
   - 响应时间：立即启动应急预案

### Why - 需求原因
1. **当前痛点**：
   - 现有风险监控依赖人工定时查询，存在15分钟数据延迟
   - 2023年Q2曾因风险发现延迟导致单日亏损200万元
   - 无法满足监管对实时风控的要求

2. **业务价值**：
   - 将风险发现时间从15分钟缩短至1秒以内
   - 预计每年避免潜在损失500万元以上
   - 满足监管合规要求，避免处罚风险

3. **战略意义**：
   - 提升公司风险管理能力，增强市场竞争力
   - 为后续量化投资策略提供基础设施支撑

### Where - 使用环境
1. **物理环境**：
   - 主要使用地点：总部交易大厅（50台双屏工作站）
   - 远程访问：分支机构通过VPN接入
   - 移动端：高管可通过企业微信接收预警通知

2. **技术环境**：
   - 网络：内网环境，禁止外网直接访问
   - 认证：需通过公司统一认证平台（AD/LDAP）登录
   - 浏览器：支持 Chrome、Edge 最新两个版本
   - 屏幕分辨率：最小 1920x1080

3. **业务环境**：
   - 组织架构：向投资总监汇报，接受风控部门监督
   - 集成要求：需与现有风控平台、核心交易系统对接
   - 数据流向：从核心交易系统实时获取数据，推送预警至企业微信

### How Much - 规格约束
1. **性能指标**：
   - 风险计算延迟：< 1秒（P99）
   - 预警推送延迟：< 3秒（从触发到用户收到）
   - 数据刷新频率：每秒一次
   - 并发用户：支持100交易员同时在线

2. **容量指标**：
   - 历史数据存储：5年（约10TB）
   - 实时监控产品数：5000+ 金融产品
   - 日增量数据：约5GB

3. **可用性指标**：
   - 系统可用性：99.95%
   - 年计划停机时间：不超过4.5小时
   - 故障恢复时间（RTO）：< 30分钟

4. **资源约束**：
   - 开发周期：6个月
   - 开发团队：1名架构师 + 3名后端 + 2名前端 + 1名测试
   - 预算：开发成本 < 200万，年度运维 < 50万

### How - 使用方式
1. **日常监控流程**：
   - 交易员登录系统后，主界面展示实时风险仪表盘
   - 仪表盘显示整体风险评分、各类风险敞口、预警列表
   - 可点击进入具体产品或组合查看详细风险指标

2. **预警处理流程**：
   - 风险超标时，系统自动推送预警至：
     * Web界面弹窗（实时）
     * 企业微信（实时）
     * 邮件（5分钟内）
   - 交易员在系统中确认收到预警并记录处理措施

3. **分析查询流程**：
   - 风控人员可通过界面选择时间范围、产品范围、风险维度
   - 系统生成可视化图表和PDF报告
   - 支持导出原始数据供Excel进一步分析

4. **系统集成方式**：
   - 实时数据：通过Kafka接收核心交易系统推送
   - 预警推送：调用企业微信API发送通知
   - 数据存储：风险日志写入数据仓库供下游分析
```
