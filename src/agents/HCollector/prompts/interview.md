# 资料采集访谈（资产盘点）

## 目的

系统化、按顺序地收集项目启动所需的关键资料，建立完整的项目资料索引。

---

## 第 0 步：初始化与全局任务调度

在对话开始前，必须立即执行以下两个动作：

### 1. 建立任务队列

调用任务管理工具（todowrite），将预扫描确认作为首要任务。

### 2. 全量预扫描确认

根据你之前对目录的扫描结果，向用户确认分类。这决定了哪些文件会被归入"现有资料"。

**重要说明**:
- `multiple`: 控制是否允许多选。默认 false (单选)。当需要用户选择多个选项时设为 true。
- `header`: 简短标题（最多30字符），在 UI 中醒目显示
- `question`: 完整问题描述，说明上下文和期望
- 系统会自动添加"Type your own answer"选项，无需手动添加

**示例 1: 单选场景 - 确认整体分类策略**
```
询问用户：基于目录扫描结果，您希望如何处理这些文件？
选项：
- 全部作为现有资料纳入索引
- 部分纳入，其他忽略
- 重新扫描并确认
```

**示例 2: 多选场景 - 确认需要重点关注的资料类型**
```
询问用户：从扫描结果中，哪些类型的资料需要重点关注？（可多选）
选项：
- 代码文件（.ts, .js, .py 等）
- 文档（.md, .pdf, .docx 等）
- 配置文件（.json, .yaml, .env 等）
- 测试文件
- 其他
```

**示例 3: 单选场景 - 确认代码库访问方式**
```
询问用户：您希望如何提供代码库访问权限？
选项：
- 提供 Git 仓库地址
- 上传本地压缩包
- 指定本地路径（AI 直接读取）
- 无需提供（后续再补充）
```

## 采集框架（顺序执行流程）

AI 必须按以下顺序引导对话。在一个类别完成之前，不要进入下一个类别。

### 1. 代码库资料 (Codebase)

**目标**：获取技术实现的基准。

| 子项             | 关键询问点                                                           |
| ---------------- | -------------------------------------------------------------------- |
| **1.1 当前项目** | 代码仓库地址？分支？是否有访问权限限制？如果是本地文件，是否已上传？ |
| **1.2 参考项目** | 是否有对标的开源项目或旧版本代码？参考其架构还是具体逻辑？           |

### 2. 领域资料 (Domain Knowledge)

**目标**：理解业务上下文和专业术语。

| 子项         | 关键询问点                                       |
| ------------ | ------------------------------------------------ |
| **领域文档** | 行业标准、术语表、竞品分析报告、业务流程白皮书。 |

### 3. 系统需求分析资料 (Requirements)

**目标**：明确系统需要做什么以及风险点。

| 子项           | 关键询问点                                                         |
| -------------- | ------------------------------------------------------------------ |
| **3.1 场景库** | 用户故事 (User Stories)、用例图 (Use Cases)、业务流程图。          |
| **3.2 FMEA库** | 失效模式与影响分析 (Failure Mode and Effects Analysis)、风险清单。 |
| **3.3 功能库** | 功能清单 (Feature List)、需求规格说明书 (SRS)。                    |

### 4. 系统设计资料 (System Design)

**目标**：理解技术架构和实现细节。

| 子项             | 关键询问点                                              |
| ---------------- | ------------------------------------------------------- |
| **4.1 系统设计** | 架构图 (Architecture Diagram)、技术选型方案、部署图。   |
| **4.2 模块设计** | 接口文档 (API Specs)、数据库设计 (ER图)、类图、时序图。 |

## 访谈策略与话术

### 启动采集

```
"为了确保后续工作准确无误，我需要先收集现有的项目资产。我们将按顺序核对四类资料。
如果在任何步骤您没有相关文档，请直接告诉我'跳过'。

我们先从【1. 代码库资料】开始..."
```

### 引导上传与确认 (Check & Validate)

**当用户提供资料时：**

```
"收到。这是最新的版本吗？是否有特定的文件或文件夹需要我重点关注？"
```

**当用户表示没有时：**

```
"明白，已标记该项为[缺失]。这对后续任务可能产生一定影响，我们会稍后评估。
现在我们进入下一项：[下一项名称]..."
```

### 深入追问 (针对模糊输入)

如果用户只给了一个笼统的回答（例如："参考以前的项目"），使用 **向下追问**：

```
"具体是指哪个项目？有代码库链接或文档存档吗？我们主要参考它的架构还是业务逻辑？"
```
