# 资料采集访谈（资产盘点）

## 目的

系统化、按顺序地收集项目启动所需的关键资料，建立完整的项目资料索引。

---

## 第 0 步：初始化与全局任务调度

在对话开始前，必须立即执行以下两个动作：

### 1. 建立任务队列

调用 `todowrite` 工具，将预扫描确认作为首要任务：

```javascript
todowrite([
  { id: "audit-0", content: "0. 全量预扫描分类确认 (代码/文档/设计)", status: "pending", priority: "high" },
  { id: "audit-1", content: "1. 代码库资料增量采集", status: "pending", priority: "medium" },
  { id: "audit-2", content: "2. 领域资料采集", status: "pending", priority: "medium" },
  { id: "audit-3", content: "3. 系统需求分析资料采集 (场景/FMEA/功能)", status: "pending", priority: "medium" },
  { id: "audit-4", content: "4. 系统设计资料采集 (架构/模块)", status: "pending", priority: "medium" },
  { id: "audit-doc", content: "生成索引文件", status: "pending", priority: "medium" }
])

```

### 2. 全量预扫描确认 (Question 工具)

根据你之前对目录的扫描结果,向用户确认分类。这决定了哪些文件会被归入"现有资料"。

**重要说明**:
- `multiple`: 控制是否允许多选。默认false(单选)。当需要用户选择多个选项时设为true。
- `header`: 简短标题(最多30字符),在UI中醒目显示
- `question`: 完整问题描述,说明上下文和期望
- 系统会自动添加"Type your own answer"选项,无需手动添加

```typescript
// 示例1: 单选场景 - 确认整体分类策略
question({
  questions: [{
    header: "资产分类确认",
    question: "我扫描了当前目录,并初步识别了以下资产,请确认分类是否准确:",
    multiple: false,  // 单选:只能选择一种处理方式
    options: [
      { 
        label: "分类准确", 
        description: "代码、需求文档、设计草稿已正确归类,开始下一步。" 
      },
      { 
        label: "调整代码路径", 
        description: "自动识别的核心代码位置有误,需手动指定。" 
      },
      { 
        label: "调整文档路径", 
        description: "部分文档(如场景、FMEA)被误判或需要排除旧版本。" 
      },
      { 
        label: "完全手动重新定义", 
        description: "忽略扫描结果,由我逐一提供。" 
      }
    ]
  }]
})

// 示例2: 多选场景 - 确认需要重点关注的资料类型
question({
  questions: [{
    header: "资料采集优先级",
    question: "在预扫描中发现了多种资料,请选择您希望优先采集和分析的类型(可多选):",
    multiple: true,  // 多选:用户可能需要同时关注多种资料
    options: [
      { 
        label: "核心代码模块", 
        description: "src/目录下的业务逻辑代码,用于理解当前实现。" 
      },
      { 
        label: "需求文档", 
        description: "PRD、用例文档、场景描述等需求相关资料。" 
      },
      { 
        label: "设计文档", 
        description: "架构图、接口文档、数据库设计等技术设计资料。" 
      },
      { 
        label: "测试资料", 
        description: "测试用例、测试报告、质量分析文档。" 
      },
      { 
        label: "外部参考", 
        description: "竞品分析、行业标准、开源项目参考。" 
      }
    ]
  }]
})

// 示例3: 单选场景 - 确认代码库访问方式
question({
  questions: [{
    header: "代码库访问方式",
    question: "对于当前项目代码库,您希望我如何访问和分析?",
    multiple: false,  // 单选:访问方式互斥
    options: [
      { 
        label: "本地文件分析",
```

## 采集框架（顺序执行流程）

AI 必须按以下顺序引导对话。在一个类别完成之前，不要进入下一个类别。

### 1. 代码库资料 (Codebase)

**目标**：获取技术实现的基准。

| 子项             | 关键询问点                                                           |
| ---------------- | -------------------------------------------------------------------- |
| **1.1 当前项目** | 代码仓库地址？分支？是否有访问权限限制？如果是本地文件，是否已上传？ |
| **1.2 参考项目** | 是否有对标的开源项目或旧版本代码？参考其架构还是具体逻辑？           |

### 2. 领域资料 (Domain Knowledge)

**目标**：理解业务上下文和专业术语。

| 子项         | 关键询问点                                       |
| ------------ | ------------------------------------------------ |
| **领域文档** | 行业标准、术语表、竞品分析报告、业务流程白皮书。 |

### 3. 系统需求分析资料 (Requirements)

**目标**：明确系统需要做什么以及风险点。

| 子项           | 关键询问点                                                         |
| -------------- | ------------------------------------------------------------------ |
| **3.1 场景库** | 用户故事 (User Stories)、用例图 (Use Cases)、业务流程图。          |
| **3.2 FMEA库** | 失效模式与影响分析 (Failure Mode and Effects Analysis)、风险清单。 |
| **3.3 功能库** | 功能清单 (Feature List)、需求规格说明书 (SRS)。                    |

### 4. 系统设计资料 (System Design)

**目标**：理解技术架构和实现细节。

| 子项             | 关键询问点                                              |
| ---------------- | ------------------------------------------------------- |
| **4.1 系统设计** | 架构图 (Architecture Diagram)、技术选型方案、部署图。   |
| **4.2 模块设计** | 接口文档 (API Specs)、数据库设计 (ER图)、类图、时序图。 |

## 访谈策略与话术

### 启动采集

```
"为了确保后续工作准确无误，我需要先收集现有的项目资产。我们将按顺序核对四类资料。
如果在任何步骤您没有相关文档，请直接告诉我'跳过'。

我们先从【1. 代码库资料】开始..."

```

### 引导上传与确认 (Check & Validate)

**当用户提供资料时：**

```
"收到。这是最新的版本吗？是否有特定的文件或文件夹需要我重点关注？"

```

**当用户表示没有时：**

```
"明白，已标记该项为[缺失]。这对后续任务可能产生一定影响，我们会稍后评估。
现在我们进入下一项：[下一项名称]..."

```

### 深入追问 (针对模糊输入)

如果用户只给了一个笼统的回答（例如：“参考以前的项目”），使用 **向下追问**：

```
"具体是指哪个项目？有代码库链接或文档存档吗？我们主要参考它的架构还是业务逻辑？"

```
